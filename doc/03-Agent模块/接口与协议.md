# AI Agent 模块：接口与协议

## 1. 服务形态

- **本地服务**：以 stdio、Unix socket 或 HTTP 形式对外提供；主仓库与侧边栏 AI 对话通过主仓库后端连接该服务。
- 推荐：Remote-SSH 下无端口转发时，**stdio** 或 **Unix socket**（工作区或临时目录下）更易用；若需跨机调试可选用 **HTTP**（localhost）。

## 2. 会话与记忆 ID 约定

- **session_id**：每个对话会话一个 ID，由主仓库或本模块生成，用于关联短期记忆与历史消息。
- **memory_id** / **chunk_id**：长期记忆或知识库片段的唯一标识，用于更新、删除与召回。
- 具体字段名与格式在实现时定（如 UUID 或时间戳+序列），并在本文档或附录中列出。

## 3. 请求/响应格式（概要）

以下为逻辑分类，具体 JSON schema 在实现时维护。

### 3.1 对话

- **请求**：`{ "action": "chat", "session_id": "...", "message": "用户输入", "context": { "file": "...", "symbol": "...", "selection": "..." }, "workflow_id": "默认或内置/自定义 ID（可选）", "agent_id": "内置或自定义 Agent ID（可选，本次对话默认使用该 Agent 作为生成端）" }`
- **响应**：流式时逐 chunk 返回文本或 SSE；非流式时 `{ "reply": "..." }`。可选附带 `memory_updated`、`cited_chunks` 等。编排器按 `workflow_id` 选择工作流；若指定 `agent_id` 则生成步骤派发到该 Agent（内置或自定义），见 §3.8。

### 3.2 记忆

- **写入**：`{ "action": "memory_write", "session_id": "...", "content": "...", "tags": [...], "summary": "..."（可选，未传则服务端自动摘要）, "parent_id": "..."（可选，树状结构）, "key_point": true（可选，记为对话要点） }` → `{ "memory_id": "..." }`。长期记忆存储**全文（content_full）+ 摘要（summary）**，支持**树状结构**（parent_id、tree_id、level）；策略见《记忆管理策略》§3。
- **召回**：`{ "action": "memory_recall", "session_id": "...", "query": "...", "limit": N, "scope": "session"|"global"（可选）, "include_key_points": true（可选）, "only_key_points": true（可选）, "tree_id": "..."（可选，树状召回）, "node_id": "..."（可选） }` → `{ "chunks": [ { "memory_id", "summary", "content_full"（可选）, "tags", "created_at", "path": [...]（可选，树路径） } ] }`。召回综合**语义**（向量匹配 summary）、**时间**、**容量**；对话要点可优先或单独召回；详见《记忆管理策略》§3.4、§3.5。
- **短期记忆**：无单独 action；由 chat 自动维护；**每 N 轮**对之前对话再次摘要（滚动摘要），见《记忆管理策略》§2.6；保留策略见§2。

### 3.3 知识库

- **更新**（若由主仓库/解析结果驱动）：`{ "action": "kb_update", "source": "codexray"|"document", "items": [ { "id": "...", "text": "...", "meta": {} } ] }` → `{ "ok": true }`；若 source 为 document，可传 `"path": "文件路径", "format": "pdf"|"docx"|"doc"|"md"` 触发**文档解析**（含 OCR）与**树状摘要建树**，详见《知识库树状结构与节点设计》。
- **检索**：多由 Agent 内部在生成回复时调用；若需单独接口：`{ "action": "kb_search", "query": "...", "limit": N, "tree_context": true }` → `{ "results": [ { "id": "节点id", "text": "摘要或原文", "meta": {}, "path": ["根标题", "章", "节"], "node": { "id", "tree_id", "parent_id", "level", "summary", "source": { "doc_id", "format", "page" } } } ] }`；支持树状检索时每条结果可带 **path**（从根到该节点的路径）与 **node** 结构。
- **树结构**（可选）：`{ "action": "kb_tree_get", "tree_id": "...", "node_id": "..." }` → `{ "tree": [ 节点列表或嵌套 ] }`，返回整棵树或从某节点起的子树，用于树状展示或上下文组装；节点结构见《知识库树状结构与节点设计》§3。

### 3.4 Skills（开放标准）

- **列出技能**：`{ "action": "skills_list" }` → `{ "skills": [ { "id": "...", "name": "...", "description": "...", "input_schema": {} } ] }`
- **调用技能**：多由编排器在对话中按需调用；若需单独接口：`{ "action": "skill_invoke", "skill_id": "...", "input": {} }` → `{ "result": ... }`
- 技能定义格式采用开放标准（如 SKILL.md 或约定 JSON schema），从配置目录加载。

### 3.5 MCP 服务

- **连接/列出 MCP 能力**：`{ "action": "mcp_list_tools" }` 或连接时发现 → `{ "tools": [ { "name": "...", "description": "...", "inputSchema": {} } ] }`；`{ "action": "mcp_list_resources" }` → `{ "resources": [...] }`
- **调用 MCP 工具**：编排器或对话中调用：`{ "action": "mcp_call_tool", "name": "...", "arguments": {} }` → `{ "content": [...] }`
- **读取 MCP 资源**：`{ "action": "mcp_read_resource", "uri": "..." }` → 资源内容。MCP 协议细节以 [Model Context Protocol](https://spec.modelcontextprotocol.io/) 为准，本服务实现为 MCP 客户端。

### 3.6 大模型设置

- **必须支持设置大模型**。配置来源任选其一或同时支持：
  - **启动时**：通过配置文件（如工作区或配置目录下的 `llm.json`）或环境变量（如 `CODEXRAY_LLM_MODEL`、`CODEXRAY_LLM_ENDPOINT`、`CODEXRAY_LLM_API_KEY`）传入；主仓库在启动 Agent 前写入配置文件或将配置写入环境。
  - **运行时**：`{ "action": "llm_set", "provider": "openai"|"azure"|"ollama"|..., "model": "...", "endpoint": "...", "api_key": "..." }` → `{ "ok": true }`；`{ "action": "llm_get" }` → `{ "provider": "...", "model": "...", "endpoint": "..." }`（api_key 不返回）。
- **对话时**：若支持按会话切换模型，chat 请求可带可选字段 `"llm_override": { "model": "..." }`，否则使用当前全局配置。生成步骤（generator）始终使用当前生效的 LLM 配置调用接口。

### 3.7 工作流

- 编排器**必须支持**顺序工作流、并行工作流、监察优化工作流及其组合，以及**自定义工作流**。协议约定：
  - **列出工作流**：`{ "action": "workflow_list" }` → `{ "workflows": [ { "id": "...", "name": "...", "type": "sequential"|"parallel"|"supervision"|"custom", "description": "..." } ] }`
  - **注册自定义工作流**：`{ "action": "workflow_register", "id": "...", "definition": { "nodes": [...], "edges": [...], "steps": [...] } }` → `{ "ok": true }`（definition 格式在实现时定，如 DSL 或 JSON 图）。
  - **chat 中指定工作流**：见 3.1，请求中可选 `"workflow_id": "..."`。
- 内置工作流至少包含：默认对话（可为顺序+监察）、纯顺序、纯并行、监察优化等；自定义工作流由用户或配置注册后可通过 `workflow_id` 调用。

### 3.8 自定义 Agent

- **必须支持自定义 Agent**。除内置 Agent（如 retriever、generator）外，用户可注册自定义 Agent，供工作流步骤按 **agent_id** 调用。
- **列出 Agent**：`{ "action": "agent_list" }` → `{ "agents": [ { "id": "...", "name": "...", "type": "builtin"|"custom", "description": "..." } ] }`，返回内置与已注册的自定义 Agent。
- **注册自定义 Agent**：`{ "action": "agent_register", "id": "...", "name": "...", "type": "subprocess"|"script"|"llm_prompt"|...（实现时定）, "definition": { ... } }` → `{ "ok": true }`。definition 格式在实现时定，可包含：subprocess 时 `command`、`args`、`env`；script 时 `interpreter`、`path`；llm_prompt 时 `system_prompt`、可选 `model_override` 等。亦可从**配置目录**加载（如 `agents/*.json`），与 agent_register 等效。
- **chat / 工作流中指定 Agent**：chat 请求可选 `"agent_id": "..."`（表示本次对话默认使用该 Agent 作为生成端）；自定义工作流的 definition 中某步骤可指定 `"agent_id": "..."`，编排器将该步骤派发到对应内置或自定义 Agent 执行。
- **执行约定**：自定义 Agent 的输入/输出与内置 Agent 一致（如接收 prompt/context，返回文本或流式 chunk）；subprocess/script 类通过 stdin/stdout 或约定协议与主进程通信，实现时在《Agent模块详细功能与架构设计》中约定。

## 4. 错误与流式

- **错误**：响应中 `{ "error": "code", "message": "..." }` 或 HTTP 4xx/5xx + body。
- **流式**：建议用 SSE 或逐行 JSON（如 NDJSON），每行一个 chunk；主仓库将 chunk 顺序转交侧边栏 UI 展示。

## 5. 与主仓库的对接

- 主仓库在扩展激活时启动或连接 Agent 服务（根据配置：stdio 子进程、socket 或 HTTP 客户端）。
- 侧边栏用户发送消息 → 主仓库构造上述请求（含当前文件/符号上下文）→ 发送给 Agent 服务 → 将回复或流式 chunk 通过 VSCode 消息传给侧边栏 Webview。
- 可选「将当前回答或解析结果写入知识库」：主仓库发送 kb_update 或等价 action，或由 Agent 在对话中自行写入，依实现约定。
