# AI Agent 模块：模块功能说明

## 1. 模块定位

- **子仓库**：AI Agent 管理模块（如 `codexray-agent`）。
- **技术**：Rust 实现。
- **职责**：Agent 记忆管理、多 Agent 编排、持久化更新记忆、本地知识库；对外提供本地服务供主仓库与侧边栏 AI 对话 UI 使用。

## 2. 功能列表

| 需求编号 | 功能 | 说明 |
|----------|------|------|
| 3.1 | 记忆管理 | Agent 的短期/长期记忆结构、读写 API；与 UI 对接时由主仓库转发请求，本模块负责存储与召回 |
| 3.1 | 多 Agent 编排 | 编排器必须支持**顺序工作流、并行工作流、监察优化工作流**及其**相互组合**与**自定义工作流**；多 Agent 分工与调度按所选工作流执行；**必须支持自定义 Agent**（用户可定义/注册自己的 Agent，如通过配置、定义文件或协议注册，编排器在工作流中可调度内置与自定义 Agent）；对外呈现为统一对话或任务接口 |
| 3.1 | 持久化更新记忆 | 记忆落盘、增量更新、版本或快照策略（如按会话或按时间） |
| 3.1 | 本地知识库 | 与 CodeXray 解析结果/文档的索引与检索（如 RAG），供 Agent 生成回复时使用；可对接解析引擎 DB 或主仓库提供的摘要/片段 |
| 3.2 | Skills 开放标准 | 支持符合开放标准的技能（Skills）：从约定目录或配置加载技能定义（如 SKILL.md）、列出可用技能、在对话/任务中按需调用技能；技能可扩展 Agent 能力（如代码操作、文档生成、工作流） |
| 3.2 | MCP 服务 | 支持 **Model Context Protocol（MCP）**：作为 MCP 客户端连接外部 MCP 服务器，发现并调用其提供的 tools、resources；使 Agent 能使用文件系统、数据库、API 等 MCP 暴露的能力，与生态工具互通 |
| 3.3 | 设置大模型 | **必须支持设置大模型**：用户可配置当前使用的 LLM，包括模型标识（如 gpt-4、claude、qwen、ollama 模型名）、API 端点、API Key（或由环境变量/密钥管理提供）、可选 provider 类型（OpenAI 兼容、Azure、Ollama 本地等）；配置在会话或全局生效，由主仓库写入后由本模块读取，或通过协议在运行时更新 |

## 3. 记忆模型（约定要点）

- **短期记忆**：**仅绑定当前会话**（session_id）；存储对话轮次与当前上下文；**每 N 轮**对之前对话做**再次摘要**（滚动摘要，可选写入长期）；保留策略：最近 N 轮 FIFO、会话结束清理。详见《记忆管理策略》§2、§2.6。
- **长期记忆**：**跨会话持久化**；结构为**全文 + 摘要**（写入时做摘要处理），并采用**树状结构**（parent_id、tree_id、level）；召回除**时间、容量**外必须考虑**语义**（向量匹配 summary）；**对话要点**单独或带 tag 保存，供**超长期**使用、不参与或最后参与淘汰。详见《记忆管理策略》§3。
- **读写 API**：主仓库或协议层调用「写入记忆」「按会话/query/标签召回记忆」等接口；请求/响应格式在《接口与协议》§3.2 定义；策略细节（容量、淘汰、在对话中的使用）在《记忆管理策略》中细化。

## 4. 编排模型（约定要点）

- 编排器**必须支持**以下三类工作流，并支持其**相互组合**与**自定义工作流**：
  - **顺序工作流（Sequential）**：步骤按固定顺序依次执行（如 理解 → 检索 → 生成），前一步输出作为后一步输入。
  - **并行工作流（Parallel）**：多个步骤或子任务并发执行，结果汇总（如多路检索并行、多 Agent 并行生成后合并）。
  - **监察优化工作流（Supervision / Optimization）**：由监察者（Supervisor）或优化步骤对中间/最终结果进行评审、打分、修正或重试（如生成 → 监察评分 → 按需优化/重生成）。
  - **组合**：上述三种可嵌套组合（如「并行检索 + 顺序生成 + 监察优化」）。
  - **自定义工作流**：用户或配置可定义新工作流（如通过 DSL、JSON 或注册步骤图），编排器解析并执行。
  - **自定义 Agent**：除内置 Agent（如检索、生成）外，**必须支持自定义 Agent**。用户可通过**配置目录**（如约定格式的 JSON/YAML 定义文件）或**协议**（agent_register）注册自定义 Agent；定义可包含类型（如 subprocess、script、llm_with_prompt）、名称、描述、入口命令或 system_prompt 等；编排器在工作流步骤中按 **agent_id** 分发到内置或已注册的自定义 Agent 执行，与内置 Agent 同一套输入/输出约定。
- 对外可暴露为单一「对话」或「任务」入口，请求中可指定 `workflow_id` 或使用默认工作流；工作流中可指定某步骤使用 `agent_id`（内置或自定义）；内部由本模块负责工作流解析、Agent 注册表与调度、状态管理。

## 5. 本地知识库

- **数据来源**：CodeXray 解析结果（如符号、调用关系、类关系摘要）、用户文档或注释等；**文档**支持 **PDF、Word（.doc/.docx）、Markdown** 等常用格式。
- **文档解析**：使用**开源库**实现正文与结构提取；对**图片/扫描件中的文字**使用 **OCR**（如 Tesseract 或 Rust OCR 开源库）进行识别，详见《知识库树状结构与节点设计》。
- **树状摘要与树状检索**：使用**本地部署模型**对资料库进行**分层摘要**，形成**树状知识结构**（根=文档摘要，中间层=章节/段落摘要，叶=原文片段）；检索支持**树状搜索结构**（层级/路径/语义检索结合，结果可带路径与子树上下文）。建树流程与节点结构见《知识库树状结构与节点设计》。
- **能力**：索引（节点表 + 向量/全文索引）、树 CRUD、RAG 检索（含树状上下文）；可选由主仓库推送摘要或本模块从配置路径拉取文档并解析建树。
- **存储**：与记忆分离或独立后端（如 SQLite + 节点表与索引、或嵌入式向量库）；树节点结构、索引与检索约定见《知识库树状结构与节点设计》与《接口与协议》§3.3。

## 6. 与主仓库的边界

- 主仓库负责：建立与 Agent 服务的连接（stdio 或 socket/HTTP）、将侧边栏用户输入与上下文（当前文件/符号）转发给本模块、将回复（含流式）转交给侧边栏 UI、可选地触发「写入知识库」等。
- 本模块负责：记忆的读写与持久化、多 Agent 编排、知识库检索与更新、生成回复；Skills 的加载与调用、MCP 客户端的连接与 tools/resources 调用；不负责 VSCode UI 渲染。
- **大模型设置**：主仓库在扩展设置中提供 codexray.llmProvider / llmModel / llmEndpoint / llmApiKey；启动 Agent 时将这些配置写入 Agent 的配置文件或环境变量，或通过协议 `llm_set` 下发；本模块读取配置并供生成步骤使用，**必须支持设置大模型**。
- **Skills / MCP**：主仓库可通过协议请求列出技能、调用技能、列出/调用 MCP 工具或读取资源；或由 Agent 在对话中自动使用。技能目录与 MCP 服务器配置由本模块读取（工作区或配置路径）。
- **Remote-SSH**：本模块以二进制或源码形式在远程主机运行，主仓库在远程扩展主机上连接本地（远程机上的）Agent 服务，行为与本地一致；Skills 与 MCP 在远程侧同样可用。
