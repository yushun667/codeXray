# 记忆管理策略：短期与长期

本文档细化**短期记忆**与**长期记忆**的管理策略，包括作用范围、存储内容、保留与淘汰、**召回维度（时间/容量/语义）**、**摘要与树状结构**、**定期对话摘要**、**对话要点（超长期）**，以及与对话/召回的配合方式。与 [模块功能说明](模块功能说明.md) §3、[接口与协议](接口与协议.md) §3.2、[Agent模块详细功能与架构设计](Agent模块详细功能与架构设计.md) 中 memory 模块配套使用。

---

## 1. 策略总览

| 维度 | 短期记忆 | 长期记忆 |
|------|----------|----------|
| **作用范围** | 单会话（与 session_id 强绑定） | 跨会话（可选按 session_id 关联或全局）；**树状结构**（节点含 parent_id、level） |
| **存储内容** | 对话轮次（user/assistant）、当前上下文；**滚动摘要**（每 N 轮对之前对话再做摘要） | **全文 + 摘要**（content_full、summary）；用户或 Agent 显式写入；**对话要点**单独或带 tag 存为超长期 |
| **持久化** | 不落盘或仅会话存活期内落盘（可选） | 必须落盘；入库前对 content 做**摘要处理**，结构含全文与摘要 |
| **保留策略** | 按轮次上限 FIFO、会话结束清理；**每 N 轮**对之前对话做一次摘要（可写入长期或仅更新会话内滚动摘要） | 按**时间/容量/语义**综合召回；可选按条数/时间/LRU 淘汰；**对话要点**不淘汰或最低优先级淘汰 |
| **召回方式** | 按 session 取最近 N 轮 + 当前 context；可选带「会话内滚动摘要」 | **语义**（向量相似度）+ **时间**（近期优先）+ **容量**（limit）；按 session_id、query、tags、scope；长期为**树状**时可按路径/层级召回 |
| **在对话中的使用** | 每次 chat 取最近 N 轮 + context；每 N 轮触发一次对之前对话的摘要 | 可选：chat 前按 query 语义召回相关 chunks（含摘要/全文）；**对话要点**在需要时优先召回 |

## 2. 短期记忆策略

### 2.1 作用范围与绑定

- **严格按 session_id 隔离**：每个会话拥有独立的短期记忆；不同 session 之间不共享对话轮次与上下文。
- **生命周期**：与「会话」同生共死。会话由主仓库或协议层用 session_id 标识；会话结束（如用户关闭对话、超时无活动、或显式清除）时，该 session_id 下的短期记忆可被**整体清理**。
- **实现可选**：短期记忆可仅存于**进程内存**（服务重启即丢失），或**落盘到临时存储**（如 SQLite 的短期表、带 TTL 的 KV），以便服务重启后同一 session 可恢复最近轮次；若落盘，需在「会话结束」或「TTL 到期」时删除。

### 2.2 存储内容与结构

- **对话轮次（turns）**：每轮包含 `role`（user | assistant）、`content`（文本）、可选 `timestamp`；按时间顺序追加，**不可修改、不可按条删除**（仅支持「清空该会话」或「保留最近 N 轮」的裁剪）。
- **当前上下文（context）**：一份「当前」状态，可被每次请求覆盖：`file`（当前文件路径）、`symbol`（当前符号）、`selection`（当前选中文本）等；与 [接口与协议](接口与协议.md) §3.1 的 context 一致。
- **滚动摘要（rolling summary，可选）**：在会话内维护一份「对当前会话已发生对话的摘要」；每经过 **K 轮**（K 可配置，如 5～10）对**之前**的对话（含上一段滚动摘要）再次调用摘要模型生成新摘要，用于拼入 prompt 时压缩长上下文；见 §2.6。
- **不存储**：长期记忆的 chunks、知识库内容、技能/MCP 调用结果；这些由各自模块管理，编排器在需要时从长期记忆或知识库召回。

### 2.3 保留与淘汰策略

- **按轮次上限（推荐）**：每个 session 的对话轮次只保留**最近 N 轮**（N 可配置，如 20、50）；新轮追加时若超过 N，**丢弃最旧的一轮**（FIFO）。保证单会话内 prompt 长度可控，且不无限增长。
- **按 token 或字符上限（可选）**：若需更精细控制上下文长度，可改为「保留最近总 token 数 ≤ M」或「最近总字符数 ≤ M」，超出时从最旧轮开始丢弃直至满足上限。
- **会话结束即清理**：当主仓库或业务判定「会话结束」时（如调用 `short_term.clear(session_id)` 或 session 表标记为 closed），删除该 session 下全部短期记忆；若未显式结束，可配合 **TTL**（如 24 小时无新消息则视为结束并清理）。
- **不跨会话保留**：短期记忆**不**在会话结束后自动迁入长期记忆；是否将某轮结论写入长期记忆由**显式 memory_write** 或工作流中的「写入长期记忆」步骤决定。

### 2.4 容量与配置

- **单会话轮次上限 N**：建议可配置（如 `memory.short_term_max_turns`），默认 20～50；实现时写入配置或环境。
- **单条 content 长度**：可设上限（如 4K 字符），超长截断或拒绝，避免单轮过大。
- **并发会话数**：若短期记忆在内存中，需考虑多 session 的总内存占用；可设「最大活跃 session 数」，超出时淘汰最久未用的 session 的短期记忆（LRU）。

### 2.5 在对话中的使用

- **每次 chat**：编排器在生成回复前，调用 `short_term.get_recent_turns(session_id, N)` 与 `short_term.get_context(session_id)`，将「最近 N 轮 + 当前 context」拼入 prompt（或传给生成步骤）。
- **每轮结束后**：编排器调用 `short_term.add_turn(session_id, "user", user_message)` 与 `short_term.add_turn(session_id, "assistant", assistant_reply)`；若请求中带 context，调用 `short_term.set_context(session_id, context)` 更新当前上下文。
- **不自动调用 memory_recall**：短期记忆的读取由编排器在 chat 流程内固定执行；长期记忆的召回由工作流或显式 memory_recall 决定（见下文）。

### 2.6 定期对话摘要（每 N 轮）

- **触发**：每完成 **K 轮**对话（K 可配置，如 `memory.summary_every_turns` = 5 或 10），编排器或短期记忆模块触发一次「对**当前会话此前对话**的摘要」。
- **输入**：此前所有轮次（或「上一段滚动摘要 + 自上次摘要以来的新轮次」），以避免重复摘要过长文本。
- **输出**：（1）**会话内滚动摘要**：更新短期记忆中的「滚动摘要」字段，供后续 chat 拼入 prompt 时替代或补充「最近 N 轮」，以节省 token；（2）**可选写入长期记忆**：将本轮生成的摘要（及可选原文片段）写入长期记忆的**树状结构**（如作为该会话对应子树的新节点），便于跨会话召回。
- **摘要模型**：使用配置的本地摘要模型或对话 LLM 的摘要能力；与知识库树状摘要可共用或独立配置。
- **与长期树状结构的关系**：若长期记忆采用树状结构（见 §3.2），则「每 N 轮摘要」可对应在会话所属的子树下追加一个**摘要节点**（parent_id 指向上一摘要节点或会话根），形成按时间或按主题的层级，便于按路径/层级召回。

## 3. 长期记忆策略

### 3.1 作用范围与绑定

- **跨会话**：长期记忆在会话结束后仍然保留，供后续任意会话通过 `memory_recall` 或工作流内「召回」使用。
- **可选按 session 关联**：每条 chunk 可携带 `session_id`（写入时所属会话），召回时可限定「仅在该 session 下写入的 chunks」或「全局」（不按 session 过滤）；实现时在 memory_recall 的语义中约定（如 query 参数带 `scope: "session"|"global"`）。
- **持久化**：必须落盘（SQLite 或嵌入式 KV/向量库）；服务重启后仍可召回。

### 3.2 存储内容与结构

- **记忆结构：全文 + 摘要**：长期存储的每条记录同时保留**全文（content_full）**与**摘要（summary）**。写入时（memory_write 或工作流写入）对 content 做**摘要处理**（调用本地摘要模型或 LLM），将摘要写入 summary 字段，原文写入 content_full；召回时可优先用 summary 做语义匹配与展示，按需返回全文。
- **单条 chunk 字段**：`memory_id`、`session_id`（可选）、**`content_full`**（全文）、**`summary`**（摘要，用于语义检索与首屏展示）、`tags`、`created_at`、`updated_at`；若为树状节点则增加 **`parent_id`**、**`children_ids`**、**`level`**、**`tree_id`**（所属树，如按 session 或按主题建树）。
- **树状结构**：长期记忆采用**树状结构**组织（与知识库类似）：根节点可为「会话根」或「主题根」，子节点为按时间或按主题划分的摘要/要点；支持按 **parent_id / tree_id / level** 做路径检索与层级召回。建树时机：memory_write 时可指定 parent_id 或由系统自动挂到当前会话根下；每 N 轮对话摘要写入时在对应会话子树下追加节点（见 §2.6）。
- **写入来源**：**memory_write** 显式写入、工作流中「写入长期记忆」步骤、或**每 N 轮对话摘要**自动写入（可选）；写入时**必须做摘要处理**生成 summary。
- **更新与版本**：同一 `memory_id` 可 support **update**（覆盖 content_full/summary/tags，并重新生成摘要）；若需版本历史，可在 store 层保留多版本，召回时默认返回最新。

### 3.3 保留与淘汰策略

- **默认不自动淘汰**：长期记忆以「用户/Agent 显式写入」为主，默认**不**按 TTL 或条数自动删除；除非配置了淘汰策略。
- **可选淘汰策略**（实现时选其一或组合）：
  - **按条数上限**：每个 session 或全局最多保留 K 条；超出时按 **LRU** 或 **FIFO** 删除；**对话要点**（见 §3.6）不参与淘汰或最后淘汰。
  - **按时间**：超过 T 天未访问或未更新的 chunk 可归档或删除；要点可豁免。
  - **按容量**：总存储超过阈值时按优先级（如要点 > 普通长期）或 LRU 淘汰。
- **增量更新**：对已存在的 memory_id 执行 **memory_update** 时，更新 content_full 并**重新生成 summary**，不重复插入；与 [接口与协议](接口与协议.md) §3.2 一致。

### 3.4 召回策略：时间、容量与语义

- **召回需同时考虑三个维度**：
  - **语义**：在 **summary**（及可选 content_full）上建向量索引，按 query 的向量做**语义检索**（相似度排序），优先返回与当前问题语义相关的 chunks。
  - **时间**：在语义或关键词初筛后，可按 **created_at / updated_at** 做时间加权或过滤（如优先近期），或按「最近访问」排序。
  - **容量**：通过 **limit** 限制返回条数；可选按 token 总上限截断，保证拼入 prompt 不超长。
- **memory_recall(session_id, query, limit, scope)**：在长期记忆的 **summary**（及可选 content_full）上做**语义检索**（向量）+ 可选**关键词**（全文）；按 session_id、scope（session|global）、tags 过滤；结果按相似度与时间综合排序后取 limit 条；返回 chunks 可含 **summary**（必）与 **content_full**（可选，按需带全文）。
- **树状召回（可选）**：若指定 tree_id 或 node_id，可返回从该节点起的**子树**或**路径**（从根到该节点），便于按主题/会话层级组装上下文。
- **在 chat 中的使用**：若工作流含「召回长期记忆」步骤，编排器在生成前调用 `memory_recall`（query 可由用户 message 或理解步骤生成），将召回的 **summary** 或 content_full 拼入 prompt；**对话要点**可在召回时优先或单独召回（见 §3.6）。

### 3.5 对话要点（超长期）

- **用途**：保存对话中需要**超长期使用**的要点（如用户明确说「请记住…」、工作流中标记为「要点」的结论、或 Agent 提炼的跨会话重要信息），与普通长期记忆区分，**不参与或最后参与淘汰**，便于长期保留与优先召回。
- **存储**：可与长期记忆共用表结构，通过 **tags 含 "key_point"** 或**独立表/命名空间**（如 `memory_key_points`）实现；字段同样含 **content_full + summary**、session_id、tags、parent_id（若纳入树状结构）、created_at。
- **写入**：通过 **memory_write** 且 `tags` 包含 `"key_point"` 或单独 action **memory_write_key_point**；工作流中可增加「将本结论记为要点」步骤。
- **召回**：**memory_recall** 时可选参数 **include_key_points: true** 或 **only_key_points: true**；语义/时间/容量逻辑与普通长期一致，但**优先返回要点**或单独返回要点列表，供 prompt 优先使用。
- **淘汰**：要点**不**参与按条数/时间/LRU 的自动淘汰，或仅在全局容量告急时按最久未访问的要点淘汰；实现时在 store 层标记 `is_key_point` 并在淘汰逻辑中排除或最后处理。

### 3.6 容量与配置

- **单条 content_full / summary 长度**：可设上限（如 8K / 512 字符），超长拒绝或截断；写入时摘要由模型生成，长度可控。
- **单会话或全局条数上限**：可选配置（如 `memory.long_term_max_chunks_per_session`、`memory.long_term_max_chunks_global`），配合淘汰策略使用；**要点**不计入或单独上限。
- **向量维度与索引**：在 **summary**（及可选 content_full）上建向量索引，用于**语义召回**；与知识库向量索引可共用或分离；约定嵌入模型与维度。
- **树状结构**：配置 `memory.long_term_use_tree`（是否启用树）、`memory.tree_id_by_session`（是否按 session 建树）；建树与路径召回见 §3.2、§3.4。
- **定期摘要**：`memory.summary_every_turns`（每 K 轮触发对话摘要）、是否写入长期树节点，见 §2.6。

## 4. 短期与长期的协同

- **写入时机**：短期记忆每轮 chat 结束自动追加；**每 N 轮**触发对之前对话的摘要（滚动摘要 + 可选写入长期树节点）。长期记忆在 **memory_write**、工作流「写入长期记忆」、以及**每 N 轮摘要写入**时写入；写入时**必须做摘要处理**，结构含 **全文 + 摘要**；**对话要点**通过 tags 或单独接口写入，不参与或最后参与淘汰。
- **读取时机**：短期每次 chat 取最近 N 轮 + context，可选带滚动摘要；长期通过 **memory_recall**（**语义 + 时间 + 容量**）召回，返回含 summary 与可选 content_full；**要点**可优先或单独召回。
- **一致性**：长期记忆采用**树状结构**与**全文+摘要**；召回时综合**语义、时间、容量**，树状时可按路径/层级召回；要点与普通长期共用 schema 或独立表，淘汰时要点豁免或最后淘汰。

## 5. 与接口、实现的对应

- **接口与协议** §3.2：memory_write 支持 **content**（存为 content_full）及可选 **summary**（若未传则服务端自动摘要）；响应 memory_id；memory_recall 支持 **query、limit、scope**，召回时**语义优先**（向量匹配 summary），结果含 **summary** 与可选 **content_full**；可选 **include_key_points / only_key_points**、**tree_id / node_id** 做树状召回。**对话要点**写入可用 tags 含 "key_point" 或单独 action。
- **Agent 详细设计** §4.4～§4.6：短期实现**每 N 轮摘要**（滚动摘要 + 可选写长期）；长期与 store 实现**全文+摘要**、**树状结构**（parent_id、tree_id、level）、**摘要处理**（写入时调用摘要模型）；store 支持 **is_key_point** 或独立要点表，淘汰时排除或最后淘汰要点；召回 API 支持语义（向量）+ 时间 + 容量，以及树状与要点优先。
- **配置项建议**：`memory.short_term_max_turns`、`memory.summary_every_turns`（每 K 轮对话摘要）、`memory.short_term_ttl_hours`（可选）、`memory.long_term_max_chunks_per_session`、`memory.long_term_eviction`、`memory.long_term_use_tree`、`memory.tree_id_by_session`；要点不淘汰或单独配置。
