# 解析引擎：数据库设计

## 1. 选型与目标

- **选型**：建议 **SQLite**，单文件、无需独立服务，便于本地与 Remote-SSH 环境部署，主仓库或解析引擎均可直接读写。
- **目标**：持久化解析结果，支持按符号、文件、调用关系、类关系、数据流、控制流等高效查询（索引设计见下）。

## 2. 表结构（概要）

以下为逻辑表设计，具体字段与类型在实现时细化。

### 2.1 元数据与工程

- **project**：工程信息（id, root_path, compile_commands_path, 等）。
- **file**：文件路径与工程关联（id, project_id, path, 等）。
- **parse_run**（**历史解析记录**）：每次解析（全量或增量）一条记录。字段：id, project_id, **started_at, finished_at**, **mode**（full / incremental）, **files_parsed**（本次解析文件数）, **status**（running / completed / failed）, 可选 error_message；用于历史列表与审计。
- **parsed_file**（懒解析 + 增量用）：记录已解析过的文件及**用于增量判断的元数据**。字段：project_id, file_id 或 path, **parse_run_id** 关联本次解析, **file_mtime** 或 **content_hash**（用于增量时判断是否变更）, parsed_at。懒解析下用于判断某次查询是否需按需解析；**增量更新**时：对比当前文件 mtime/hash 与表中记录，仅对已变更文件重新解析，并更新/删除该 file 在 symbol、call_edge、class、global_var、cfg 等表中的旧数据后写入新结果。

### 2.2 符号与调用链

- **symbol**：函数/方法等符号（id, **usr**, name, qualified_name, kind, **def_file_id, def_line, def_column** 定义位置，**def_line_end, def_column_end** 定义范围终点；可选声明位置；可选 decl_*；以 def 为准用于跳转，定义范围供高亮/选中）。
- **call_edge**：调用边（caller_id, callee_id, call_site_file_id, call_site_line, call_site_column, **edge_type**：direct / via_function_pointer），用于构建调用图。直接调用一条边对应一个 callee；**经函数指针的间接调用**可为一条边对应一个”可能 callee”，即同一调用点可有多条 call_edge（caller 与 call_site 相同、callee 不同），以表示**函数指针所有可能的调用链**。
  - **UNIQUE 约束**：`UNIQUE(caller_id, callee_id, call_file_id, call_line, call_column)`，防止跨 TU 重复插入同一调用关系（多个 TU 可能看到相同的调用点，如头文件中的内联函数调用）。写入时使用 `INSERT OR IGNORE` 跳过重复行。
- **function_pointer_call**（可选，与 call_edge 二选一或互补）：若希望显式建模“一调用点 → 多可能目标”，可增加此表（call_site 标识, caller_id, possible_callee_ids 或多条 (caller_id, call_site, callee_id)），便于查询“该函数指针调用点所有可能走向”。
- 索引：symbol(**usr** 唯一), symbol(name, def_file_id, qualified_name)，call_edge(caller_id, callee_id)，call_edge(call_site_*, edge_type)。

### 2.3 类关系

- **class**：类/结构体（id, **usr**, file_id, name, qualified_name, **def_file_id, def_line, def_column** 定义位置，**def_line_end, def_column_end** 定义范围终点，等）。
- **class_relation**：类间关系（parent_id, child_id, relation_type：inheritance / composition / dependency）。
- **class_member**：成员（class_id, symbol_id 或 type 等）。
- 索引：class(name, file_id)，class_relation(parent_id, child_id, relation_type)。

### 2.4 全局变量与数据流

- **global_var**：全局变量（id, **usr**, **def_file_id, def_line, def_column** 定义位置，**def_line_end, def_column_end** 定义范围终点, file_id, name, 等）。
- **data_flow_edge**：数据流边（var_id, reader_id/writer_id, 读写位置等）。
- 索引：global_var(name, file_id)，data_flow_edge(var_id, reader_id, writer_id)。

### 2.5 控制流（Protobuf 文件存储）

> **v2 重要变更**：控制流数据量极大（llvm-project 全量解析达 1.24 亿节点 + 1.07 亿边），SQLite 行存储导致数据库文件高达 11GB。为解决存储问题，CFG 数据改为以 **Protobuf 二进制文件**方式存储，SQLite 仅保留索引。

#### 2.5.1 存储结构

- **cfg_index**（SQLite 索引表）：
  - 字段：`symbol_id INTEGER NOT NULL UNIQUE REFERENCES symbol(id)`，`pb_path TEXT NOT NULL`
  - `pb_path` 为相对于数据库目录（`.codexray/`）的路径
  - 索引：`cfg_index(symbol_id UNIQUE)`
  - 作用：将函数符号映射到其 CFG 数据所在的 pb 文件

- **cfg/ 目录**（文件系统）：每个函数对应一个独立的 `.pb` 文件。
  - **路径格式**：`cfg/<hash[0:2]>/<hash>.pb`，其中 `<hash>` 为函数 USR 的 **FNV-1a 64-bit** 哈希值（16 位十六进制），`<hash[0:2]>` 取前两位做子目录分片，防止单目录文件过多。
  - **哈希算法**：FNV-1a 64-bit（初始值 `14695981039346656037`，乘数 `1099511628211`），纯 C++ 实现，无外部依赖。
  - **示例路径**：`cfg/3a/3a7f2b9c01e8d4f5.pb`
  - **文件内容**：Protobuf 二进制序列化的 `FunctionCfg` 消息。

#### 2.5.2 Proto 定义

文件位置：`parser/proto/cfg.proto`

```proto
syntax = "proto3";
package codexray.cfg;

message CfgNode {
  int32  block_id    = 1;  // Clang CFGBlock::getBlockID()，函数内唯一
  string file_path   = 2;  // 源文件路径（首条语句所在文件）
  int32  begin_line  = 3;
  int32  begin_col   = 4;
  int32  end_line    = 5;
  int32  end_col     = 6;
  string label       = 7;  // "entry" / "exit" / terminator 类型名称
}
message CfgEdge {
  int32  from_block  = 1;  // 源基本块 block_id
  int32  to_block    = 2;  // 目标基本块 block_id
  string edge_type   = 3;  // "unconditional" / "true" / "false"
}
message FunctionCfg {
  string            function_usr = 1;  // 函数 Clang USR
  repeated CfgNode  nodes        = 2;
  repeated CfgEdge  edges        = 3;
}
```

#### 2.5.3 写入流程（WriteCfg）

1. 按 `function_usr` 将 `CfgNodeRow` 和 `CfgEdgeRow` 分组。
2. 对每个函数：
   - 从 `usr_sym_cache_` 查找 `symbol_id`（需先 `WriteSymbols` 完成）。
   - 计算 USR 的 FNV-1a 64-bit 哈希（`Fnv1a64Hex`），得到 16 位十六进制字符串。
   - 构造相对路径 `cfg/<hash[0:2]>/<hash>.pb`，绝对路径为 `<db_dir>/<rel_path>`。
   - 构建 `codexray::cfg::FunctionCfg` Protobuf 消息，填入节点和边。
   - `fs::create_directories` 创建分片子目录。
   - `SerializeToString` → 以二进制模式写入 `.pb` 文件。
   - 在 SQLite `cfg_index` 表执行 `INSERT OR REPLACE INTO cfg_index(symbol_id, pb_path)` 记录索引。

#### 2.5.4 查询流程（QueryControlFlow）

1. 通过 `ResolveSymbolId` 将函数 USR 或名称解析为 `symbol_id`。
2. 查询 `cfg_index` 表：`SELECT pb_path FROM cfg_index WHERE symbol_id=?`。
3. 拼接绝对路径：`<db_dir>/<pb_path>`。
4. 读取 `.pb` 文件全部内容，调用 `FunctionCfg::ParseFromString` 反序列化。
5. 将 Protobuf 节点/边转换为查询结果结构体（`CfgNodeQ`、`CfgEdgeQ`），其中节点 `id` 和 `block_id` 均使用 Clang `CFGBlock::getBlockID()`。

#### 2.5.5 增量更新时的清理（DeleteDataForFile）

增量更新删除文件旧数据时，需同时清理 CFG：

1. 查询该文件关联的 pb 文件路径：`SELECT ci.pb_path FROM cfg_index ci JOIN symbol s ON ci.symbol_id=s.id WHERE s.def_file_id=?`。
2. 删除文件系统上对应的 `.pb` 文件（`fs::remove`）。
3. 从 `cfg_index` 表删除索引记录：`DELETE FROM cfg_index WHERE symbol_id IN (SELECT id FROM symbol WHERE def_file_id=?)`。

#### 2.5.6 实测存储效果

| 项目 | SQLite 行存储 | Protobuf 文件存储 | 压缩率 |
|------|--------------|------------------|--------|
| llvm-project（2935 TU，1.24 亿 CFG 节点） | ~11GB | ~2.8GB | **~75%** |

Protobuf 二进制编码紧凑，且每个函数独立文件便于增量更新和按需读取；查询时仅需一次 `cfg_index` 查询 + 读取单个 pb 文件，性能与原 SQL `SELECT WHERE symbol_id=?` 方案相当。

## 3. 查询场景示例

- **解析历史**：查 parse_run 表，按 started_at 降序，供主仓库展示历史解析记录列表。
- 按 USR 或符号名查调用者/被调用者：查 symbol（含 usr、定义位置）+ call_edge；函数指针调用会得到多条边（同一 call_site 对应多个 possible callee）。
- 查某函数指针调用点的所有可能调用链：查 call_edge where edge_type = via_function_pointer 且 call_site 匹配，或查 function_pointer_call 表。
- 按类名查继承/组合关系：查 class + class_relation。
- 按全局变量查读写链：查 global_var + data_flow_edge。
- 按函数查 CFG：先查 `cfg_index` 获取 `pb_path`，然后读取对应 `.pb` 文件并反序列化 `FunctionCfg` 消息。

## 4. 与主仓库的约定

- 数据库文件路径由主仓库或配置传入（如工作区 `.codexray/codexray.db`）；解析引擎负责创建/迁移 schema 与写入，主仓库或解析引擎提供只读查询 API 供扩展使用。
- Remote-SSH 下 DB 位于远程工作区路径，查询同样在远程执行。

## 5. 已修复的查询 Bug

### 5.1 ResolveSymbolId 对匿名命名空间方法的 `def_file_id=NULL` 问题

**问题描述**：

匿名命名空间（anonymous namespace）中定义的函数和方法，其 Clang USR 包含源文件名（如 `c:IntrinsicEmitter.cpp@N@(anonymous namespace)@S@...`），但在多 TU 合并时，如果定义所在的 TU 不是最后一个写入 DB 的 TU，`def_file_id` 可能为 `NULL`。这导致 `ResolveSymbolId` 通过 `name + def_file_id JOIN file` 的方式无法找到这些符号。

**根因分析**：

`DbWriter::GetOrInsertSymbolId` 使用 `UPSERT` 语义（`INSERT ... ON CONFLICT(usr) DO UPDATE`），但 `def_file_id` 使用 `COALESCE(excluded.def_file_id, def_file_id)` 策略。当某个 TU 的 symbol 仅有声明而无定义时，`def_file_id` 为 `NULL`；若该 TU 后于包含定义的 TU 写入，`COALESCE` 不会用 `NULL` 覆盖已有值，此处逻辑正确。但对于匿名命名空间的方法，由于其 USR 绑定到文件，跨 TU 不会合并，只有定义所在 TU 写入一次——而如果该 TU 的 `def_in_tu_file` 判断失误（如头文件中定义被误判），`def_file_id` 将保持为 `NULL`。

**修复方案**：

在 `ResolveSymbolId`（`db/reader/reader.cc`）中增加第三级 fallback：当 `name + def_file_id JOIN` 查询失败时，提取 `file_path` 的基本文件名（basename），在 `symbol` 表中查找 `name` 匹配且 `usr LIKE '%<basename>%'` 的符号。这利用了匿名命名空间 USR 中包含源文件名的特性。

```sql
-- 三级查询策略
-- 1. USR 精确匹配
SELECT id FROM symbol WHERE usr = ?
-- 2. name + def_file_id JOIN file
SELECT s.id FROM symbol s JOIN file f ON s.def_file_id=f.id
  WHERE s.name=? AND f.path LIKE ? LIMIT 1
-- 3. name + USR 包含文件名 fallback（匿名命名空间）
SELECT id FROM symbol WHERE name=? AND usr LIKE ? LIMIT 1
```

**实现位置**：`parser/src/db/reader/reader.cc`，`ResolveSymbolId` 函数。

### 5.2 call_edge 跨 TU 重复插入

**问题描述**：多个 TU 解析时，头文件中的内联函数调用等会被多个 TU 重复看到，导致 `call_edge` 表中出现大量重复行（caller_id、callee_id、调用位置完全相同），浪费存储并影响查询结果。

**修复方案**：为 `call_edge` 表添加 `UNIQUE(caller_id, callee_id, call_file_id, call_line, call_column)` 约束，写入时使用 `INSERT OR IGNORE` 跳过重复行。

**实现位置**：`parser/src/db/schema.h`（建表语句）、`parser/src/db/writer/writer.cc`（INSERT 语句）。

### 5.3 删除函数指针 Stage 2 保守枚举

**问题描述**：函数指针分析的 Stage 2 通过签名兼容性枚举所有可能的 callee（`CollectFunctionsByType`、`CollectMethodsByMemberPtrType`、`FunctionProtoCompatible`），产生大量误报边，严重干扰调用图的可读性和准确性。

**修复方案**：删除 `CollectFunctionsByType`、`CollectMethodsByMemberPtrType`、`FunctionProtoCompatible` 三个函数。`GetPossibleCallees` 不再做签名兼容性枚举，仅依赖 Stage 1 数据流分析（VarDecl 赋值追踪）。策略为**宁可漏报也不误报**。

**实现位置**：`parser/src/ast/function_pointer/analyzer.cc`、`analyzer.h`。
