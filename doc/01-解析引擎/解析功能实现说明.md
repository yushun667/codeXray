# 解析功能实现说明

本文档详细说明 CodeXray 解析引擎（codexray-parser）如何实现 C/C++ 代码的解析：从 compile_commands 到 AST 分析再到 SQLite 落库的完整流程与关键实现细节。

---

## 1. 整体流程概览

解析入口为 CLI 子命令 `parse`，由 `main.cc` 解析参数后调用 `RunParse(ParseOptions, ParseSummary*)`（`driver/orchestrator.cc`）。流程分为以下阶段：

1. **初始化**：打开/创建 SQLite 数据库，执行 Schema 与迁移，确保 project 与 file 表存在。
2. **加载编译命令**：从 `compile_commands.json` 加载 TU 列表，得到每个源文件对应的编译参数（`compile_args`）与工作目录。
3. **确定待解析 TU**：根据 `--incremental` / `--lazy` 与 `--priority-dirs` 决定本次要解析的 TU 子集（全量、仅变更、或仅优先目录）。
4. **并行解析**：线程池对每个 TU 调用一次「单 TU 处理函数」：运行 Clang 前端得到 AST，在同一 AST 上依次执行四种分析，结果写入进程内容器。
5. **汇总与写库**：按 USR 合并多 TU 的 symbol，将 symbols、call_edges、classes、relations、global_vars、data_flow_edges 批量写入 DB；CFG 数据序列化为 Protobuf 文件后写入 `cfg/` 目录，并在 `cfg_index` 表记录索引。更新 parsed_file 与 parse_run。

以下按模块说明各步的实现方式。

---

## 2. 编译命令加载（compile_commands）

**位置**：`parser/src/compile_commands/load.cc`、`load.h`。

**职责**：读取 JSON Compilation Database（[Clang 文档](https://clang.llvm.org/docs/JSONCompilationDatabase.html)），为每个编译单元生成一条 `TUEntry`。

**实现要点**：

- **输入**：`project_root` 与可选路径 `path_to_cc`；默认 `path_to_cc = project_root + "/compile_commands.json"`。
- **JSON 解析**：使用 **nlohmann/json** 解析；要求根节点为数组，每项为对象且包含 `directory`、`file`，以及 `arguments`（数组）或 `command`（字符串）。
- **每条记录**：
  - `directory` + `file` 拼成源文件的绝对路径（`MakeAbsolute`），并做路径规范化（`NormalizePath`）。
  - 编译参数来源：
    - 若有 `arguments` 数组：直接逐项 `get<std::string>()` 得到 `args`。
    - 否则若有 `command` 字符串：调用 `SplitCommandLine` 按空格与引号规则拆成 argv 风格列表（只保留第一个 `;` 之前的部分）。
  - **TruncateToFirstJob**：Clang 一次只接受一个编译 job；在第一个 `;` 或第二个 `-cc1` 处截断，避免多 job 导致错误。
  - 若末尾为 `-c` + 源文件路径，则去掉该源文件参数（与 `FixedCompilationDatabase` 只接受一个源文件一致）。
  - **AppendFsyntaxOnly**：若参数中尚无 `-fsyntax-only`，则追加；解析仅做语法/语义分析，不生成目标文件，加快解析。
- **输出**：`std::vector<TUEntry>`。`TUEntry` 包含 `source_file`（规范化绝对路径）、`compile_args`（字符串向量）、`working_directory`（规范化目录）。

**懒解析与优先目录**：`SplitByPriorityDirs(all_tus, project_root, priority_dirs, &priority, &rest)` 按相对路径前缀将 TU 分为「优先」与「其余」，供 `--lazy` 时首轮只解析优先目录。

## 3. Driver 与 TU 列表确定（orchestrator）

**位置**：`parser/src/driver/orchestrator.cc`。

**RunParse 逻辑**：

1. **路径与 DB**：规范化 `project_root`，确定 `compile_commands` 路径与输出 DB 路径（默认 `project_root/.codexray/codexray.db`）。
2. **连接与 Schema**：`Connection::Open(db_path)`，`EnsureSchema(conn.Get())`（建表 + symbol 的 decl_* 列迁移）。
3. **Project**：`DBWriter::EnsureProject(project_root, cc_path)` 得到 `project_id`。
4. **加载 TU**：`LoadCompileCommands(project_root, cc_path)` 得到 `all_tus`；若为空则返回 `kCompileCommands`。
5. **确定 tus_to_run**：
   - **增量模式**（`--incremental`）：`GetChangedFiles` 得到变更文件列表；`RemoveDataForFiles` 删除这些文件在 DB 中的旧 symbol/call_edge/class/… 数据。若无变更且历史上从未解析过，则按「全量」解析；若无变更且已有历史，则 `tus_to_run` 为空；若有变更，则只保留「源文件在变更集合中」的 TU。
   - **非增量**：若 `--lazy` 且配置了 `priority_dirs`，则 `tus_to_run = priority`（仅优先目录）；否则 `tus_to_run = all_tus`。
6. **历史记录**：`InsertParseRun(project_id, mode)` 插入一条 `parse_run`（status=running），得到 `run_id`。
7. **线程池与处理器**：见下一节。
8. **写库与收尾**：将合并后的 symbols、edges、classes、… 写入 DB，更新 `parsed_file`，提交事务，`UpdateParseRun` 为 completed，并上报进度 100%。

---

## 4. 线程池调度（scheduler/pool）

**位置**：`parser/src/scheduler/pool.cc`、`pool.h`。

**职责**：对 `tus_to_run` 做 TU 级并行，每个 TU 只被一个线程处理一次。

**实现**：

- **并行度**：若调用方传入 `parallel == 0`，则 `DefaultParallel()` = `max(1, hardware_concurrency() - 2)`（可配置为其它策略）；且不超过 TU 数量。
- **任务队列**：使用 `std::queue<size_t>` 存 TU 下标，`std::mutex` 保护取任务。
- **Worker**：每个线程循环：加锁取一个下标 `idx`，解锁后执行 `processor(tus[idx])`；若成功则 `++done_count` 并调用 `on_progress(done, total)`。
- **进度**：开始时 `on_progress(0, total)`；每完成一个 TU 调用一次；结束时 `on_progress(total, total)`。Driver 将 `total` 设为 `tus.size() + 1`（多一步「写 DB」），从而解析完成时进度约 98%，写库完成后再 100%。

**约束**：同一 TU 不会被多线程同时处理；处理器内部可再使用锁访问共享容器（如 `symbol_by_usr`、`all_edges`）。

---

## 5. 单 TU 处理管道（processor 与 combined action）

每个 TU 在 processor 中依次完成：**分配 file_id → 跑一次 Clang 解析并做四种分析 → 填 file_id → 合并到全局容器**。

### 5.1 分配 file_id

在持有 `db_mutex` 下调用 `writer.EnsureFile(project_id, tu.source_file)`，得到该源文件在 `file` 表中的 `file_id`，供本 TU 所有结果的 `def_file_id` / `call_site_file_id` / `file_id` 等使用。

### 5.2 一次 Clang 解析、四种分析（ast/combined）

**目标**：每个 TU 只调用一次 Clang 前端（只解析一次源码），在同一棵 AST 上依次执行四种分析，以降低解析耗时。

**入口**：`RunAllAnalysesOnTU(tu, &cg_out, &cr_out, &df_out, &cf_out)`（`ast/combined/action.cc`）。

**步骤**：

1. **工作目录**：`WorkingDirForTU(tu)`：优先用 `tu.working_directory`，否则从 `tu.source_file` 取目录并规范化。
2. **编译数据库**：`FixedCompilationDatabase(dir, tu.compile_args)`，用该 TU 的编译参数；`ClangTool(*db, [tu.source_file])` 只编译这一个源文件。
3. **头文件与 -fsyntax-only**：通过 `GetClangIncludeEnvExtraArgs()` 取得进程内缓存的 `-resource-dir`、`-isystem` 列表（来自 `clang++ -E -x c++ - -v` 与 `clang++ -print-resource-dir` 的解析结果），用 `getInsertArgumentAdjuster(..., BEGIN)` 插入到命令行最前；`-fsyntax-only` 已在 load 阶段加入 `compile_args`。
4. **运行工具**：`ClangTool::run(&CombinedActionFactory(...))`。Clang 内部会：
   - 解析该源文件（词法、语法、语义），生成一棵 AST；
   - 调用 `FrontendAction::CreateASTConsumer` 得到 `CombinedConsumer`；
   - 在解析完成后调用 `ASTConsumer::HandleTranslationUnit(ASTContext&)`。
5. **HandleTranslationUnit 内**：对**同一** `ASTContext` 依次调用：
   - `RunCallGraphAnalysis(ctx, cg_out)`
   - `RunClassRelationAnalysis(ctx, cr_out)`
   - `RunDataFlowAnalysis(ctx, df_out)`
   - `RunControlFlowAnalysis(ctx, cf_out)`  
   即四种分析共享同一份 AST，不再重复解析。

各分析模块提供两类接口：

- **RunXxxOnTU(tu, out)**：独立使用时自己建 `ClangTool`、跑一次前端、再在 AST 上跑分析（当前主路径未用）。
- **RunXxxAnalysis(ctx, out)**：仅在已有 AST 上做遍历，供 combined 在 `HandleTranslationUnit` 里调用。

## 6. 四种 AST 分析的实现

四种分析均基于 Clang 的 `RecursiveASTVisitor` 或 CFG API，在**同一 AST** 上分别遍历，产出结构化记录（symbol、call_edge、class、class_relation、global_var、data_flow_edge、cfg_node、cfg_edge）。

### 6.1 调用图（call_graph）

**位置**：`parser/src/ast/call_graph/action.cc`。

**目的**：收集所有函数/方法的**符号**（含定义与声明位置、USR、kind）以及**调用边**（caller → callee，含调用点位置）。

**实现**：

- **Visitor**：`CallGraphVisitor` 继承 `RecursiveASTVisitor<CallGraphVisitor>`。
  - **TraverseFunctionDecl**：对每个 `FunctionDecl` 均尝试收集（含仅声明、含定义、含隐式、含类外定义），**仅当无法生成 USR 时跳过**，不因 implicit 或 getLocation() 无效而跳过。
    - 用 `index::generateUSRForDecl` 生成 USR，用 `SymbolFromDecl` 填 `SymbolRecord`（name、qualified_name、kind、def_*、decl_*、def_in_tu_file、decl_in_tu_file）。
    - **kind**：根据类型区分为 `function`、`method`（CXXMethodDecl）、`constructor`（CXXConstructorDecl）、`destructor`（CXXDestructorDecl）。
    - **定义/声明位置**：定义用 `getBeginLoc`/`getEndLoc`；声明用 `getLocation()`；是否在当前 TU 主文件用 `SourceManager::getFileID(loc) == getMainFileID()` 判断，供 driver 填 `def_file_id`/`decl_file_id`。
    - 将 symbol 加入 `out_->symbols`。若当前是**定义**，则把 `current_function_usr_` 压栈并设为该函数 USR，以便后续 `VisitCallExpr` 使用。
  - **TraverseCXXDeductionGuideDecl / TraverseCXXMethodDecl / … / TraverseCXXConversionDecl**：Clang 的 `RecursiveASTVisitor` 对上述类型会走各自的 Traverse，而不会走 `TraverseFunctionDecl`，若只重写后者会导致未被收集。因此显式将 **CXXDeductionGuideDecl**（C++17 推导指南）、**CXXMethodDecl**、**CXXConstructorDecl**、**CXXDestructorDecl**、**CXXConversionDecl** 五种 Traverse 委托到 `TraverseFunctionDecl(D)`，从而所有函数形态（含推导指南、成员函数、构造/析构/转换运算符）均会进入 symbol 与 call_edge。symbol 的 kind 对应为 `deduction_guide`、`method`、`constructor`、`destructor`、`function`。
  - **VisitCallExpr**：若当前处于某函数定义内（`current_function_usr_` 非空），则构造 `CallEdgeRecord`：caller_usr = `current_function_usr_`，call_site 行列为 `E->getBeginLoc()`。**被调解析**：若为 `CXXMemberCallExpr` 则用 `getMethodDecl()` 取 callee；若为析构则 edge_type = "destructor"，若为虚函数则 edge_type = "virtual"；否则若被调为 `DeclRefExpr` → `FunctionDecl` 则 edge_type = "direct"；否则若 `E->getDirectCallee()` 非空（含 CUDA kernel 等扩展）则取该 callee；否则通过 `GetPossibleCallees` 写 edge_type = "via_function_pointer" 的边。
  - **VisitCXXConstructExpr**：构造函数调用（如 `T x(args);`、`return T(args);`、临时对象）产生 caller → 构造函数 边，edge_type = "constructor"。
- **函数指针**：`ast/function_pointer/analyzer.cc` 的 `GetPossibleCallees` 仅通过 Stage 1 数据流分析（VarDecl 赋值追踪，如 `fp = &func`）确定可能的 callee USR 列表。不做签名兼容性的保守枚举，策略为宁可漏报也不误报。

**能解析的调用类型**（会生成 caller → callee 边）：

| 类型 | Clang AST / 说明 | 实现方式 / edge_type |
|------|------------------|----------------------|
| 直接函数调用 | `CallExpr`，callee 为 `DeclRefExpr` → `FunctionDecl` | 取该 USR，edge_type = "direct" |
| 成员函数调用 | `CXXMemberCallExpr`（如 `obj.f()`、链式调用） | `getMethodDecl()`，edge_type = "direct" |
| **显式析构调用** | `obj.~T()` 等，同为 `CXXMemberCallExpr`，getMethodDecl() 为析构 | edge_type = "destructor" |
| **虚函数调用** | 同上，getMethodDecl() 为 virtual 方法 | edge_type = "virtual"（记录静态声明的 callee） |
| **构造函数调用** | `CXXConstructExpr`（`T x(args);`、临时对象、return T(args)） | `VisitCXXConstructExpr` → getConstructor()，edge_type = "constructor" |
| **CUDA / 扩展直接调用** | `CUDAKernelCallExpr` 等有直接 callee 的 CallExpr 子类 | `getDirectCallee()` 回退，edge_type = "direct"（若需可后续标为 "cuda_kernel"） |
| 运算符重载 | `CXXOperatorCallExpr` | 多为 DeclRefExpr 分支，edge_type = "direct" |
| 函数指针 / 可调用对象 | 非直接 callee | `GetPossibleCallees`，edge_type = "via_function_pointer" |

**当前仍部分或未覆盖**：模板/依赖名（UnresolvedLookupExpr 等）、伪析构 `p.~int()`（无实际析构）、Block/Lambda 经泛型调用的部分。

**输出**：`CallGraphOutput`：`symbols`（SymbolRecord 列表）、`edges`（CallEdgeRecord 列表）。Driver 会按 USR 合并多 TU 的 symbol（保留更完整的 def/decl），并统一填 `def_file_id`/`decl_file_id`（仅当 `def_in_tu_file`/`decl_in_tu_file` 为真时用本 TU 的 file_id）。

### 6.2 类关系（class_relation）

**位置**：`parser/src/ast/class_relation/action.cc`。

**目的**：收集类/结构体的**定义**（CXXRecordDecl）及其**继承关系**（基类列表）。

**实现**：

- **Visitor**：`ClassRelationVisitor`。
  - **VisitCXXRecordDecl**：仅处理**有定义**且非 implicit、location 有效的声明。对每个这样的类：用 USR、name、qualified_name、def 起止行列填 `ClassRecord` 并加入 `classes`；遍历 `D->bases()`，对每个基类取 `CXXRecordDecl` 的 USR，写一条 `ClassRelationRecord`（parent_usr、child_usr、relation_type = "inheritance"）到 `relations`。

**输出**：`ClassRelationOutput`：`classes`、`relations`。Driver 为每条记录填 `file_id`/`def_file_id`（本 TU 的 file_id）。

### 6.3 数据流（data_flow）

**位置**：`parser/src/ast/data_flow/action.cc`。

**目的**：收集**文件作用域全局变量**及其**读/写点**（哪个函数读、哪个函数写）。

**实现**：

- **Visitor**：`DataFlowVisitor`。
  - **TraverseFunctionDecl**：仅对定义压栈当前函数 USR，用于后续读/写点归属。与 call_graph 相同，**TraverseCXXMethodDecl / TraverseCXXConstructorDecl / TraverseCXXDestructorDecl / TraverseCXXConversionDecl** 委托到 `TraverseFunctionDecl`，使成员函数参与数据流（其内对全局变量的读写会被正确归属）。
  - **VisitVarDecl**：仅处理具有全局存储且非局部/参数的变量（`hasGlobalStorage()` 且非 local/param），记入 `global_vars` 并加入 `global_usr_set_`。
  - **VisitDeclRefExpr**：若引用的是全局变量（USR 在 `global_usr_set_`），则写一条 data_flow 边：var_usr、reader_usr = 当前函数、读点位置。
  - **VisitBinaryOperator**（赋值）、**VisitUnaryOperator**（++/--）：若左端/操作数为全局变量，则写一条边：var_usr、writer_usr = 当前函数、写点位置。

**输出**：`DataFlowOutput`：`global_vars`、`edges`（DataFlowEdgeRecord）。Driver 填 `def_file_id`/`file_id`。

### 6.4 控制流（control_flow）

**位置**：`parser/src/ast/control_flow/action.cc`。

**目的**：对每个**有体**的函数构建 CFG，输出**基本块节点**与**块间控制流边**。

**实现**：

- **BuildCFGForFunction**：对给定 `FunctionDecl`，用 `CFG::buildCFG(FD, FD->getBody(), &ctx, opts)` 得到 CFG。遍历所有 `CFGBlock`，为每个块生成 `CfgNodeRecord`（symbol_usr、block_id、kind="block"、行号等）；再遍历块的后继，生成 `CfgEdgeRecord`（from_node_index、to_node_index、edge_type="control"）。节点在输出数组中的下标用于边的 from/to。
- **Visitor**：`ControlFlowVisitor` 的 **VisitFunctionDecl** 对每个有体的函数定义调用 `BuildCFGForFunction`。与 call_graph 同理，RecursiveASTVisitor 对成员函数会走 **VisitCXXMethodDecl** 等而非 VisitFunctionDecl，因此显式实现 **VisitCXXMethodDecl / VisitCXXConstructorDecl / VisitCXXDestructorDecl / VisitCXXConversionDecl** 并委托到 `VisitFunctionDecl(D)`，使成员函数的 CFG 也被构建。

**输出**：`ControlFlowOutput`：`nodes`、`edges`。Driver 为节点填 `file_id`，并将边的 from/to 下标加上当前 TU 在全局 cfg_nodes 中的起始下标后写入 `all_cfg_edges`。

**存储方式（v2 Protobuf）**：CFG 数据不再写入 SQLite 表行，而是按函数分组后序列化为 Protobuf 二进制文件（每个函数独立一个 `.pb` 文件），详见《数据库设计》§2.5。SQLite 仅保留 `cfg_index(symbol_id, pb_path)` 索引表。查询时先查索引获取 pb 文件路径，读取并反序列化 `FunctionCfg` 消息。

## 7. 结果汇总与按 USR 合并（driver 内）

Processor 内对**符号**做按 USR 的合并，其它数据按 TU 追加：

- **symbol**：用 `std::unordered_map<std::string, SymbolRecord> symbol_by_usr`。每收到一个 symbol：若 USR 未见过则插入；若已存在则只在不覆盖已有信息的前提下更新 def_* 或 decl_*（例如本 TU 有定义而之前只有声明，则补全 def_*；同理 decl_*）。kind 上若已有 "function" 而本次为 method/constructor/destructor 则升级为后者。
- **edges / classes / relations / global_vars / data_flow_edges**：直接 `push_back` 到 `all_*` 容器。
- **cfg**：nodes 直接追加；edges 的 from/to 下标加上当前 `all_cfg_nodes.size()` 的基准后再追加，保证指向全局 node 数组中的正确位置。
- **file_id**：在 processor 内根据本 TU 的 `file_id` 为各记录的 def_file_id、decl_file_id、call_site_file_id、file_id 等统一赋值（symbol 的 def/decl 仅当 `def_in_tu_file`/`decl_in_tu_file` 为真时赋值）。

池子跑完后，将 `symbol_by_usr` 转成 `std::vector<SymbolRecord> all_symbols`，供写库使用。

## 8. 数据库写入（db/writer）

**位置**：`parser/src/db/writer/writer.cc`。连接层：`parser/src/db/connection.cc`。

### 8.1 WAL 与连接 PRAGMA（connection.cc）

打开 DB 后统一设置：

- **PRAGMA journal_mode=WAL**：写前日志模式，适合批量写入与并发读；一次事务写入上万条记录时减少 fsync 与锁竞争。
- **PRAGMA synchronous=NORMAL**：WAL 下足够安全，兼顾性能。
- **PRAGMA cache_size=-64000**：约 64MB 页缓存。
- **PRAGMA busy_timeout=30000**：30 秒内若被 WAL checkpoint 等阻塞则等待而非立即返回 SQLITE_BUSY。
- **PRAGMA temp_store=MEMORY**：批量 INSERT 时临时对象放内存。

### 8.2 批量事务与按 TU 缓存

- **事务边界**：整次解析只做一次 **BeginTransaction**，所有 Write* 在同一事务内完成，最后 **Commit**，实现「一次写入上万条记录」的批量事务。
- **按 TU 缓存**：边、CFG 节点等在各 TU 的 processor 内追加到 `all_edges`、`all_cfg_nodes`、`all_cfg_edges` 等，写库阶段一次性提交，不按 TU 多次提交。

### 8.3 写库顺序与批量 INSERT

在 **BeginTransaction** 与 **Commit** 之间依次：

1. **WriteSymbols**：对每个 symbol 做 `INSERT ... ON CONFLICT(usr) DO UPDATE SET ...`，合并 def/decl；用 prepared statement 循环，写后查 id 填 `usr_to_id`。
2. **WriteCallEdges**：先解析所有边为 (caller_id, callee_id, …)，缺失 callee 则查库或插入占位 symbol，结果缓存；再按批（每批 kBatchSize=100 行）多行 `INSERT INTO call_edge VALUES (?,?,?,?,?,?), ...` 执行。
3. **WriteClasses**：多行 INSERT 每批 100 行，插入后用 `last_insert_rowid()` 与行数推算本批 id 区间，填 `class_usr_to_id`。
4. **WriteClassRelations**：先转为 (parent_id, child_id, type)，再按批多行 INSERT。
5. **WriteGlobalVars**：同 WriteClasses，批量 INSERT 并回填 `var_usr_to_id`。
6. **WriteDataFlowEdges**：先解析为 (var_id, reader_id, writer_id, …)，再按批多行 INSERT。
7. **WriteCfg**（v2 Protobuf 文件存储）：
   - 按 `function_usr` 将 `CfgNodeRow` / `CfgEdgeRow` 分组（`std::unordered_map<usr, vector<ptr>>`）。
   - 对每个函数：
     1. 从 `usr_sym_cache_` 查找 `symbol_id`，若未注册则跳过。
     2. 调用 `Fnv1a64Hex(usr)` 计算 FNV-1a 64-bit 哈希（纯 C++ 实现，无外部依赖），得到 16 位十六进制字符串 `hex`。
     3. 构造 pb 文件相对路径 `cfg/<hex[0:2]>/<hex>.pb`，绝对路径 `<db_dir>/<rel_path>`。
     4. 构建 `codexray::cfg::FunctionCfg` Protobuf 消息（`parser/proto/cfg.proto`），填入 `function_usr`、所有 `CfgNode`（block_id、file_path、行列范围、label）和 `CfgEdge`（from_block、to_block、edge_type）。
     5. `fs::create_directories` 创建分片子目录，`SerializeToString` 序列化，以二进制模式写入 `.pb` 文件。
     6. 在 SQLite `cfg_index` 表执行 `INSERT OR REPLACE INTO cfg_index(symbol_id, pb_path)` 记录索引。
   - **存储效果**：llvm-project 全量解析（1.24 亿 CFG 节点 + 1.07 亿边），原 SQLite 行存储约 11GB，Protobuf 文件存储约 2.8GB，压缩约 **75%**。
   - **已替代**：原 `WriteCfgNodes` + `WriteCfgEdges` 方法已删除，由 `WriteCfg` 统一替代。
9. **UpdateParsedFile**：对本次解析涉及的每个 TU 插入或更新 `parsed_file`。

最后 **Commit**，并 **UpdateParseRun** 将对应 `parse_run` 置为 completed、写入 finished_at、files_parsed。

## 9. 辅助组件

- **Clang 头文件环境**（`common/clang_include_detector.cc`）：首次调用时执行 `clang++ -E -x c++ - -v` 与 `clang++ -print-resource-dir`，解析出系统 include 路径与 resource-dir，缓存到进程内；`GetClangIncludeEnvExtraArgs()` 返回已拼好的 `-resource-dir` / `-isystem` 参数列表，供 combined 的 ArgumentsAdjuster 使用，避免每 TU 重新构建。
- **路径**（`common/path_util.cc`）：`NormalizePath`、`MakeAbsolute`、`ToProjectRelative` 等，统一路径格式与相对路径计算。
- **增量**（`incremental/incremental.cc`）：`GetChangedFiles` 根据 `parsed_file` 的 mtime 或 content_hash 与当前文件对比得到变更列表；`DeleteDataForFile` 按 file_id 删除 symbol、call_edge、class、… 等表内相关行，**同时清理 CFG**（查询 `cfg_index` 获取该文件关联的 pb 文件路径，调用 `fs::remove` 删除物理文件，再删除 `cfg_index` 索引记录）。
- **历史**（`history/history.cc`）：`InsertParseRun`、`UpdateParseRun`、`ListRunsJson` 等，维护 `parse_run` 表与 list-runs 输出。`parse_run` 表含 `files_failed` 字段（schema 迁移 v2），`UpdateParseRun` 写入 files_parsed、files_failed。

- **进度与摘要**：parse 子命令向 stdout 输出进度 JSON 行 `{"progress": 百分比, "current_file": "..."}`（scheduler 回调带 current_file）；解析成功后输出一行摘要 JSON：`status, run_id, mode, files_parsed, files_failed, symbols`。`--verbose` 将详细日志输出到 stderr。

- **懒解析按需解析**（`driver/orchestrator.cc`）：`ParseOnDemandForQuery(project_root, db_path, compile_commands_path, file_paths, parallel, priority_dirs)` 仅解析指定源文件对应 TU，写库并更新 parsed_file；防重入。query 子命令在 `--lazy` 且目标文件未解析时先调用此接口再执行查询。

- **并行度与 --jobs**：默认并行度 `max(1, 系统核心数 - 2)`（`scheduler/pool.cc` 的 `DefaultParallel()`，cores≤2 时返回 1）。parse/query 支持 `--parallel N` 与 `--jobs N` 别名（设计 §5.6）。

## 10. 数据流小结

```
compile_commands.json
    → LoadCompileCommands (TU 列表 + 每 TU 的 compile_args)
    → [可选] GetChangedFiles + DeleteDataForFile（增量，含 cfg pb 文件清理）
    → RunTUPool(tus_to_run, parallel, processor, on_progress)
         每 TU: EnsureFile → RunAllAnalysesOnTU → 填 file_id → 合并到 symbol_by_usr / all_edges / ...
    → all_symbols = symbol_by_usr 的值
    → BeginTransaction → WriteSymbols → WriteCallEdges → ... → WriteCfg(pb 文件 + cfg_index) → UpdateParsedFile → Commit
    → UpdateParseRun(completed)
```

解析功能通过「一次 Clang 前端 + 四种 AST 遍历」和「按 USR 合并 + 批量写库」实现，在保证符号与调用/类/数据流/控制流信息完整落库的前提下，兼顾解析速率与增量更新能力。

## 11. 已修复的 Bug

### 11.1 ResolveSymbolId 对匿名命名空间方法的 `def_file_id=NULL` 问题

**问题**：查询匿名命名空间中定义的函数时，`ResolveSymbolId` 的 `name + def_file_id JOIN file` 查询失败，因为这些符号的 `def_file_id` 为 `NULL`。

**原因**：匿名命名空间的符号 USR 包含源文件名（如 `c:IntrinsicEmitter.cpp@N@(anonymous namespace)@S@...`），与文件绑定。当 `def_file_id` 因 UPSERT 合并策略遗漏而为 `NULL` 时，无法通过 `def_file_id JOIN file` 定位。

**修复**（`parser/src/db/reader/reader.cc`）：在 `ResolveSymbolId` 中增加第三级 fallback——提取查询文件路径的 basename，用 `SELECT id FROM symbol WHERE name=? AND usr LIKE '%<basename>%'` 匹配匿名命名空间的 USR，利用其 USR 中嵌入文件名的特性做定位。

### 11.2 CFG 数据迁移：SQLite 行存储到 Protobuf 文件存储

**问题**：llvm-project 全量解析产生 1.24 亿 CFG 节点 + 1.07 亿边，SQLite 行存储导致 `.db` 文件达 11GB，远超可接受范围。

**方案**：CFG 数据改为 Protobuf 二进制文件存储（每个函数独立 `.pb` 文件，FNV-1a 哈希分片目录），SQLite 仅保留 `cfg_index(symbol_id, pb_path)` 索引表。详见本文档 §8.3 第 7 项及《数据库设计》§2.5。

**效果**：llvm-project 从 ~11GB 降至 ~2.8GB，压缩约 75%。

### 11.3 call_edge 跨 TU 重复插入

**问题**：多个 TU 解析时，头文件中的内联函数调用等会被多个 TU 重复看到，导致 `call_edge` 表中出现大量重复行。

**修复**：为 `call_edge` 表添加 `UNIQUE(caller_id, callee_id, call_file_id, call_line, call_column)` 约束，写入时使用 `INSERT OR IGNORE` 跳过重复行。实现位置：`parser/src/db/schema.h`、`parser/src/db/writer/writer.cc`。

### 11.4 删除函数指针 Stage 2 保守枚举

**问题**：函数指针分析的 Stage 2 通过签名兼容性枚举（`CollectFunctionsByType`、`CollectMethodsByMemberPtrType`、`FunctionProtoCompatible`）所有可能的 callee，产生大量误报边。

**修复**：删除上述三个函数，`GetPossibleCallees` 仅依赖 Stage 1 数据流分析（VarDecl 赋值追踪）。策略为宁可漏报也不误报。实现位置：`parser/src/ast/function_pointer/analyzer.cc`、`analyzer.h`。
