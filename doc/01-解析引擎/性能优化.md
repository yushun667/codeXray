# 解析引擎性能优化

## 概述

CodeXray 解析引擎在解析大型 C/C++ 项目（如 llvm-project，2935 个编译单元）时存在性能瓶颈。本文档记录已实施的性能优化及其原理。

## 优化措施

### 1. 系统 Include 环境变量传递（最高优先级）

**问题**：每个 TU 通过 `fork()+execl()` 启动独立子进程。子进程调用 `GetClangIncludeEnv()` 时，由于 `std::call_once` 缓存是进程级的，每个子进程都会重新运行两个 `popen()` 调用探测系统头文件路径：
- `clang++ -print-resource-dir`（~50ms）
- `echo '' | clang++ -E -x c++ - -v`（~50ms）

对于 2935 个 TU，共产生 5870 次无用的子进程调用，浪费约 5-10 分钟。

**方案**：父进程探测一次，通过环境变量 `CODEXRAY_CLANG_INCLUDE_ENV` 传递给子进程。

**实现文件**：
- `parser/src/common/clang_include_detector.cc/h` — 新增 `SerializeClangIncludeEnv()`，修改 `GetClangIncludeEnv()` 优先读取环境变量
- `parser/src/scheduler/pool.cc` — 在 `RunScheduled()` 创建线程前调用 `setenv()`

**序列化格式**：使用 ASCII 0x1F (Unit Separator) 分隔，避免路径中常见的 `|`、`:` 字符冲突。格式：`resource_dir\x1Fpath1\x1Fpath2\x1F...`

**预计效果**：消除 5870 次子进程调用，节省 5-10 分钟。

### 2. 合并双事务为单事务

**问题**：每个 TU 完成后执行两个独立事务：
```
Transaction 1: BeginTransaction → WriteAll → Commit
Transaction 2: BeginTransaction → EnsureFile → UpdateParsedFile → Commit
```
每次 COMMIT 在 WAL 模式下仍需写 WAL 文件。2935 个 TU 产生 5870 次 COMMIT。

**方案**：合并为一个事务，减半 COMMIT 次数。

**实现文件**：`parser/src/driver/orchestrator.cc` — `RunParse` 和 `ParseOnDemandForQuery` 的 `on_result` 回调

**预计效果**：节省 0.5-1 分钟。

### 3. 预编译 SQL 语句

**问题**：`GetOrInsertSymbolId()`、`GetOrInsertClassId()`、`GetOrInsertGlobalVarId()`、`EnsureFile()` 等高频函数每次调用都 `sqlite3_prepare_v2()` + `sqlite3_finalize()`。235,996 个 symbols 导致约 47 万次不必要的 prepare/finalize。

**方案**：在 `DbWriter` 构造函数中预编译所有 14 条常用 SQL 语句，存储为成员变量。析构函数统一 finalize。

**实现文件**：
- `parser/src/db/writer/writer.h` — 新增 14 个 `sqlite3_stmt*` 成员、析构函数、`PrepareStatements()` 方法
- `parser/src/db/writer/writer.cc` — 所有 `Write*` 和 `GetOrInsert*` 函数改用预编译语句

**预编译语句列表**：
| 语句 | 用途 |
|------|------|
| `stmt_insert_file_` | `EnsureFile`: INSERT OR IGNORE |
| `stmt_select_file_` | `EnsureFile`: SELECT id |
| `stmt_upsert_symbol_` | `GetOrInsertSymbolId`: UPSERT |
| `stmt_select_symbol_` | `GetOrInsertSymbolId`: SELECT id |
| `stmt_upsert_class_` | `GetOrInsertClassId`: UPSERT |
| `stmt_select_class_` | `GetOrInsertClassId`: SELECT id |
| `stmt_upsert_gvar_` | `GetOrInsertGlobalVarId`: UPSERT |
| `stmt_select_gvar_` | `GetOrInsertGlobalVarId`: SELECT id |
| `stmt_insert_call_edge_` | `WriteCallEdges` |
| `stmt_insert_class_rel_` | `WriteClassRelations` |
| `stmt_insert_class_mem_` | `WriteClassMembers` |
| `stmt_insert_df_edge_` | `WriteDataFlowEdges` |
| `stmt_upsert_parsed_file_` | `UpdateParsedFile` |
| `stmt_upsert_cfg_index_` | `WriteCfg` |

**预计效果**：节省 0.5-1 分钟。

### 4. 子进程计算 mtime+hash 避免父进程重复读文件

**问题**：`on_result` 回调中调用 `GetFileMtime()` + `ComputeFileHash()` 重新读取源文件。对于 2935 个 TU，每个源文件被 Clang 分析读取一次后又被父进程读取一次，浪费 I/O。

**方案**：在 `CombinedOutput` 中新增 `source_file_mtime` 和 `source_file_hash` 字段。子进程（`parse-tu` 子命令）在分析完成后计算这两个值，随结果 JSON 返回。父进程 `on_result` 直接使用，不再重复读文件。

**实现文件**：
- `parser/src/common/analysis_output.h` — `CombinedOutput` 新增两个字段
- `parser/src/main.cc` — `parse-tu` 子命令中计算 mtime+hash
- `parser/src/scheduler/pool.cc` — JSON 序列化/反序列化新增两个字段
- `parser/src/driver/orchestrator.cc` — `on_result` 优先使用子进程计算的值

**预计效果**：节省 0.3-0.5 分钟。

### 5. Pre-fork Worker Pool + Protobuf IPC

**问题**：每个 TU 通过 `fork()+execl("parse-tu")` 启动全新子进程，存在两个瓶颈：
- **exec 加载开销**：每次 exec 加载 ~20MB 可执行文件 + 动态库（~15ms × 2935 = ~44s）
- **JSON 序列化瓶颈**：nlohmann::json 对大型 CombinedOutput 效率差（cfg_nodes 平均 42K 条/TU，每 TU ~6.3MB JSON）

**方案**：Pre-fork N 个长驻 worker 子进程，用 Protobuf 二进制帧替代 JSON IPC。

**架构**：
```
父进程 (RunScheduled)
  │ 主线程：fork+exec N 个 "parse-tu-worker" → workers[0..N-1]
  ├─ 线程 0 ← 专属 workers[0]
  │   loop: pop TU → WriteFrame(req_pb) → ReadFrame(resp_pb) → on_result
  ├─ 线程 1 ← 专属 workers[1] ...
  └─ 线程 N-1 ← 专属 workers[N-1]

Worker 子进程 (parse-tu-worker):
  loop: ReadFrame(stdin) → 反序列化 WorkerRequest → RunAllAnalysesOnTU
        → 计算 mtime/hash → 序列化 WorkerResponse → WriteFrame(stdout)
  stdin EOF → exit(0)
```

**关键设计**：
- 每线程独占一个 worker，无竞争、无需池级锁
- 仍使用 fork+exec（非纯 fork），因为 macOS 多线程 fork 不安全；但只做 N 次 exec 而非 2935 次
- Worker crash 时自动 fork+exec 替代进程（单个 TU 标记失败，继续处理队列）
- IPC 帧格式：`[4 bytes uint32_t LE 长度][N bytes protobuf 数据]`，最大帧 512MB 安全上限
- 保留 `CODEXRAY_LEGACY_FORK=1` 环境变量回退旧模式（每 TU fork+exec + JSON），用于调试和对比测试
- 保留 `parse-tu` 子命令 + JSON 序列化供独立 debug 使用

**IPC 消息定义**（`parser/proto/ipc.proto`）：
- `WorkerRequest`：3 字段（source_file, directory, arguments）
- `WorkerResponse`：ok, error, mtime, hash + 9 个 repeated 行类型 + referenced_files
- 9 个子消息 1:1 映射 `analysis_output.h` 中的结构体

**实现文件**：
| 文件 | 作用 |
|------|------|
| `parser/proto/ipc.proto` | IPC 消息定义 |
| `parser/src/scheduler/ipc_proto.h` | Protobuf 转换函数 + 帧读写声明 |
| `parser/src/scheduler/ipc_proto.cc` | 实现：CombinedOutput↔Proto 转换、WriteFrame、ReadFrame |
| `parser/src/main.cc` | 新增 `parse-tu-worker` 隐藏子命令（二进制帧循环） |
| `parser/src/scheduler/pool.cc` | 新增 Pre-fork 模式 + 保留旧模式 |
| `parser/src/scheduler/pool.h` | 注释更新（公共 API 不变） |

**Crash 恢复策略**：
1. ReadFrame 返回 EOF 或 WriteFrame 返回 EPIPE → worker 崩溃检测
2. waitpid 回收僵尸，记录信号号
3. ForkOneWorker 创建替代 worker（fork+exec 在多线程下安全）
4. 当前 TU 标记失败（不重试 — 有毒 TU 会再次 crash 新 worker）
5. 线程继续处理队列中下一个 TU

**验证结果**（llvm-project Demangle 子集，6 TU）：
- Pre-fork 模式：6/6 成功，7445 symbols，所有表记录数与旧模式完全一致
- Legacy 模式：6/6 成功，数据完全一致

**性能对比**：
| 指标 | 旧模式 (fork+exec+JSON) | 新模式 (pre-fork+protobuf) |
|------|-------------------------|----------------------------|
| 进程创建 | 2935 × exec (~15ms) = ~44s | N × exec = ~0.2s |
| 序列化体积 | ~6.3MB/TU (JSON) | ~2.1MB/TU (protobuf, ~3x 压缩) |
| 序列化速度 | nlohmann dump (慢) | protobuf SerializeToString (快) |
| 反序列化 | nlohmann parse (慢) | protobuf ParseFromString (快) |

**预计效果**：额外节省 3-8 分钟。
