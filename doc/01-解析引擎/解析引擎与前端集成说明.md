# 解析引擎与前端集成说明

本文档说明 CodeXray 扩展（前端）如何集成解析引擎（codexray-parser）：调用方式、配置项、进度与结果流、侧栏与编辑器入口。

---

## 1. 集成方式概览

- **进程模型**：扩展通过 `child_process.spawn` 调用解析器可执行体，不链接解析器库；解析器以 CLI 子命令（`parse`、`query`、`list-runs`）运行。
- **数据流**：解析器 stdout 输出 JSON 行（进度）与最终摘要 JSON；扩展解析后更新状态栏与侧栏 Webview。查询时扩展 spawn `query` 子命令，读取 stdout 的 JSON 作为图数据交给可视化面板。
- **配置与路径**：工作区根、数据库路径、compile_commands 路径、并行度等由扩展 `Config` 读取 VS Code 配置（`codexray.*`），并传入解析器参数。

---

## 2. 解析器可执行体路径

- **逻辑**：`Config.getParserPath()` 优先返回 `extensionRoot/bin/codexray-parser`（或 Windows 下 `codexray-parser.exe`）；若不存在则回退到 `extensionRoot/parser/build/codexray-parser`，便于开发时未打包也能使用已构建的 parser。
- **要求**：需在扩展安装目录下提供可执行体，或在开发时在 `parser/build/` 下完成构建（如 `cd parser && cmake -B build && cmake --build build`）。

---

## 3. 配置项（codexray.*）

| 配置键 | 类型 | 说明 |
|--------|------|------|
| `compileCommandsPath` | string | compile_commands.json 路径；可相对工作区根（如 `build/compile_commands.json`）或绝对路径。**必填**：未设置时解析会直接报错并提示用户设置。 |
| `databasePath` | string | 可选；默认未设置时使用 `<workspace>/.codexray/codexray.db`。 |
| `parserParallelism` | number | 解析并行度；0 表示 `max(1, CPU 核心数 - 2)`。对应解析器 `--parallel` / `--jobs`。 |
| `parserLazy` | boolean | 是否启用懒解析；对应 `--lazy`。 |
| `parserPriorityDirs` | string[] | 优先解析目录（相对工程根）；对应 `--priority-dirs`。 |
| `parserIncremental` | boolean | 是否默认增量解析；对应 `--incremental`。 |

---

## 4. 解析流程（parse）

1. **入口**：侧栏「解析」按钮或命令面板「CodeXray: 执行解析」→ `ParserService.parse()`。
2. **前置校验**：无工作区根则返回错误；**未配置 compileCommandsPath 时直接返回错误并提示用户设置**；配置了但文件不存在则返回「路径不存在或不可读」。
3. **参数**：`parse --project <workspaceRoot> --output-db <databasePath> --compile-commands <compileCommandsResolved> --parallel N`，以及按配置追加 `--lazy`、`--priority-dirs`、`--incremental`。
4. **进度**：解析器每完成若干 TU 向 stdout 输出一行 JSON：`{"progress": 0-100, "current_file": "..."}`。扩展解析 stdout 后取 `percent` 或 `progress`，通过 `onProgress` 回调和 `sidebarView.postToWebview({ action: 'parseProgress', percent })` 推送到侧栏与状态栏。
5. **结束**：解析器成功时最后输出一行摘要 JSON：`{"status":"ok","run_id":...,"mode":"full|incremental","files_parsed":N,"files_failed":M,"symbols":S}`。扩展将该对象作为 `ParseResult` 通过 `parseResult` 消息发给侧栏；失败时扩展根据退出码构造 `ParseResult`（status: 'error', message: ...）。
6. **UI**：状态栏显示「解析 N%」/「解析完成」/「解析失败」；侧栏（React 或内联 HTML）显示解析中进度，完成后显示「最近: X 个文件, Y 失败」或「解析失败: ...」。

---

## 5. 查询流程（query）

1. **入口**：C/C++ 编辑器中右键「查看调用链/类关系图/数据流/控制流」或对应命令 → `getCurrentSymbol()` 取当前光标/选区符号 → `ParserService.query(type, { symbol, file })`。
2. **参数**：`query --db <databasePath> --project <workspaceRoot> --type <call_graph|class_graph|data_flow|control_flow> [--depth N] [--direction both|callers|callees] [--symbol ...] [--file ...]`；call_graph 默认 depth=2，direction=both。
3. **方向感知递归**：depth>1 时，从根节点出发第一层同时查询 callers（左）和 callees（右）；后续层级沿发现方向单向递归——左链只继续查 callers，右链只继续查 callees。`--direction callers` 仅向左查调用节点，`--direction callees` 仅向右查被调用节点。
4. **输出**：解析器向 stdout 输出单行 JSON：`{ "nodes": [...], "edges": [...] }`。扩展解析后得到 `GraphData`，若无节点且无边则提示「未查询到图数据」，否则 `VisualizationProvider.openGraph(type, data)` 在编辑区打开新标签展示图。
5. **图内扩展**：用户在图内节点右键「查询调用节点」时传 `--direction callers`，「查询被调用节点」时传 `--direction callees`，返回的 nodes/edges 通过 `graphAppend` 合并到当前图。

---

## 6. 历史记录（list-runs）

- **入口**：侧栏「历史」按钮或命令「CodeXray: 查看历史解析记录」→ `ParserService.listRuns(limit)`。
- **参数**：`list-runs --db <databasePath> [--project <workspaceRoot>] [--limit N]`。
- **输出**：解析器 stdout 输出 JSON 数组：`[{ "run_id", "started_at", "finished_at", "mode", "files_parsed", "status", ... }]`，扩展解析后通过 `parseHistory` 消息发给侧栏展示。

---

## 7. 与解析引擎接口约定的对应关系

- **进度**：解析引擎接口约定 §3 进度 JSON 使用 `progress` 字段；前端兼容 `percent` 与 `progress`。
- **摘要**：解析引擎最后一行摘要包含 `status`、`run_id`、`mode`、`files_parsed`、`files_failed`、`symbols`；前端 `ParseResult` 类型与之一致并用于侧栏展示。
- **查询**：四种类型的 JSON 输出格式见 01-解析引擎/接口约定；前端 `GraphData`、`GraphNode`、`GraphEdge` 与解析器输出字段一致，可视化层按 `graphType` 做边字段映射（caller/callee、parent/child、from/to、var/reader/writer）。

---

## 8. 小结

解析引擎以前端子进程方式集成：扩展负责配置与路径、spawn 解析器、解析 stdout 的进度与摘要/查询 JSON，并驱动状态栏、侧栏解析管理、编辑器右键查询与图面板。用户需先设置 `compile_commands.json` 路径方可执行解析；解析完成后即可在 C/C++ 文件中通过右键或命令查看调用链、类关系、数据流、控制流。
