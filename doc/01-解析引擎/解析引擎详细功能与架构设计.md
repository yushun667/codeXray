# 解析引擎：详细功能与架构设计（优化版）

本文档对基于 Clang libtooling 的 C/C++ 解析引擎进行完整的功能与架构设计。本设计吸收了前期评审中的优化建议，提供了更稳健、高性能、可扩展的实现方案。所有模块不再强调代码行数限制，而是以功能完整性和工程合理性为首要目标。

本文档与 [模块功能说明.md](模块功能说明.md)、[接口约定.md](接口约定.md)、[数据库设计.md](数据库设计.md) 配套使用，三者共同构成解析引擎的完整规格。

---

## 1. 概述与设计原则

### 1.1 解析引擎职责回顾

- **核心功能**：基于 Clang libtooling 对 C/C++ 代码进行语义级分析，输出函数调用链（含函数指针完整调用链）、类关系、全局变量数据流、控制流等结构化信息。
- **标识机制**：所有符号使用 **USR** 唯一标识，并记录定义位置与声明位置。
- **持久化**：结果存入 SQLite 数据库，支持高效查询与增量更新。
- **并行能力**：默认并行度 = 系统核心数，支持懒解析（优先解析指定目录，查询时按需解析剩余）。
- **增量更新**：仅重新解析变更的编译单元（TU），合并回数据库。
- **历史记录**：记录每次解析运行的元数据，提供 `list-runs` 接口。

以上与 [模块功能说明.md] 第 2 节、[接口约定.md] 完全一致。

### 1.2 技术栈

- **语言**：C++17/20
- **解析**：Clang libtooling（FrontendAction + RecursiveASTVisitor），必要时使用 CFG 接口
- **存储**：SQLite3，表结构遵循 [数据库设计.md]
- **对外**：CLI 子命令 `parse` / `query` / `list-runs`，输出 JSON 格式

### 1.3 核心设计优化点

- **单次 AST 遍历合并多分析**：避免同一 TU 被多次解析，性能提升约 4 倍。
- **标准库头文件自动探测**：通过 `clang++ -v` 和 `-print-resource-dir` 动态获取系统头路径，解决跨平台编译问题。
- **函数指针分析策略**：分阶段实现，初期支持同一 TU 内直接赋值，后期通过数据流扩展。
- **DB 写入优化**：WAL 模式、批量事务、USR→ID 缓存、UPSERT 语义。
- **查询按需解析同步机制**：确保懒解析模式下查询能等待相关 TU 解析完成后再返回。
- **并发控制与内存管理**：限制并行任务数，支持 `--jobs` 参数，避免内存爆炸。
- **完善的错误处理与可观测性**：进度 JSON 输出、详细日志、失败文件记录、结构化错误码。

## 3. 模块详细设计

### 3.1 CLI 入口与子命令 (`main.cc`, `cli/`)

**职责**：
- 解析命令行参数（子命令 `parse` / `query` / `list-runs`）。
- 根据子命令调用对应的调度层、查询层或历史层接口。
- 解析过程中向 stdout 输出进度 JSON 行（格式：`{"progress":百分比, "current_file":"..."}`），供主仓库轮询。
- 设置全局日志级别（`--verbose` 输出到 stderr 或日志文件）。
- 错误退出时返回与 [接口约定.md] §4 一致的错误码。

**关键实现**：
- 使用 `getopt_long` 或成熟的 CLI 库（如 CLI11）解析参数。
- 进度回调由 `scheduler` 提供，CLI 层注册回调函数。
- 错误处理：捕获所有顶层异常，记录日志并返回对应错误码。

### 3.2 解析驱动 (`driver/`)

**职责**：
- 协调一次完整解析流程（全量/增量/懒解析）。
- 加载 `compile_commands.json`，获取 TU 列表。
- 若为增量模式，调用 `incremental` 获取变更文件，删除旧数据。
- 创建 `parse_run` 记录（`status=running`）。
- 根据懒解析配置和优先目录，决定首轮提交给 `scheduler` 的 TU 集合。
- 调用 `scheduler` 并行执行 TU 分析，收集结果。
- 将结果批量写入数据库（调用 `db_writer`）。
- 更新 `parsed_file` 表（记录 mtime/hash）和 `parse_run` 状态（`completed`/`failed`）。
- 提供 `ParseOnDemandForQuery(query_target)` 接口，供查询层在懒解析模式下按需解析。

**关键实现**：
- `ParseOptions` 结构体包含：project_root, compile_commands_path, db_path, parallel, lazy, priority_dirs, incremental, extra_include_dirs 等。
- 解析顺序：增量删除旧数据 → 插入 parse_run → 调度 TU → 写入 DB → 更新 parse_run。
- 按需解析时，根据目标（文件路径或 USR 所属文件）确定待解析 TU 列表，同步阻塞直到解析完成（使用 `std::future` 或回调）。

### 3.3 CompileCommands 加载 (`compile_commands/`)

**职责**：
- 读取并解析 `compile_commands.json`，生成 `TUEntry` 列表（包含源文件路径、编译参数、工作目录）。
- 将源文件路径规范化为绝对路径。
- 根据 `priority-dirs`（相对项目根）将 TU 划分为优先和剩余两类。

**关键实现**：
- 使用 JSON 库（如 nlohmann/json）解析。
- `TUEntry` 包含：source_file（绝对路径）、arguments（字符串数组）、directory（工作目录）。
- `SplitByPriorityDirs`：对于每个 priority_dir，匹配 source_file 的相对路径前缀，匹配上的归为优先，其余为剩余。

### 3.4 并行与懒解析调度 (`scheduler/`)

**职责**：
- 维护线程池，并行执行 TU 分析任务。
- 支持提交任务列表，并等待所有任务完成（或首个失败）。
- 提供进度回调（已完成任务数/总数）。
- 支持按需解析时提交单个或多个任务，并等待完成。
- 控制最大并发数（由 `--parallel` 或 `--jobs` 指定，默认系统核心数）。

**关键实现**：
- 使用 `std::thread` 和任务队列，或 `ThreadPool` 库。
- 每个任务执行 `combined::RunAllAnalysesOnTU(tu, outputs)`，结果存入线程本地存储，最后合并。
- 进度回调：每完成一个 TU 调用一次，传递当前文件路径和进度百分比。
- 内存控制：可配置最大并发任务数，避免内存溢出（默认=核心数，但允许用户通过 `--jobs` 调低）。

### 3.5 增量更新 (`incremental/`)

**职责**：
- 根据 `parsed_file` 表记录的 mtime 或 content hash，与磁盘文件对比，找出已变更的文件列表。
- 删除这些文件在数据库中的旧数据（symbol、call_edge、class、global_var、cfg 等表），保留 `parsed_file` 记录（后续会更新）。
- 提供接口供 driver 调用。

**关键实现**：
- 使用 `parsed_file` 表字段：`file_id`, `file_path`, `file_mtime`, `file_hash`, `parse_run_id`。
- 变更检测策略：若启用 `--incremental`，先读取 parsed_file，对每个文件 stat 并计算 hash（如 xxhash），与记录对比，不同则加入变更列表。
- 删除数据时按 `file_id` 删除所有关联业务表，注意外键约束（SQLite 需开启外键支持）。

### 3.6 单次 AST 分析合并 (`ast/combined/`)

**职责**：
- 对外提供 `RunAllAnalysesOnTU` 函数，内部创建一次 `ClangTool`，运行同一个 `FrontendAction`。
- 在 `FrontendAction` 的 `HandleTranslationUnit` 中，依次调用各子模块的分析函数（`call_graph::Analyze`、`class_relation::Analyze` 等），传入 ASTContext 和输出结构体。
- 汇总各子模块的输出结构体，返回给调用方。

**关键实现**：
- `RunAllAnalysesOnTU` 接受 TUEntry，返回一个聚合结构体 `CombinedOutput`（包含 symbol 列表、call_edge 列表、class 列表等）。
- 异常安全：每个子模块分析独立捕获异常，记录错误后继续执行，保证一个模块失败不影响其他模块。
- 通过 `ArgumentsAdjuster` 注入自动探测到的系统头路径（参见 3.16 公共模块）。

### 3.7 调用链分析 (`ast/call_graph/`)

**职责**：
- 遍历 AST，收集函数/方法定义和声明（symbol 表）。
- 收集直接函数调用（`CallExpr`），生成 `call_edge`（`edge_type='direct'`）。
- 当被调用者为函数指针时，委托 `function_pointer` 模块分析，生成多条 `call_edge`（`edge_type='via_function_pointer'`）。
- 输出结构体 `CallGraphOutput` 包含 `symbols` 和 `call_edges`。

**关键实现**：
- USR 生成：使用 `clang::index::generateUSRForDecl`。
- 定义位置：`def_file_id`、`def_line`、`def_column`、`def_line_end`、`def_column_end`；声明位置类似。
- 区分函数种类：`kind` 字段（`function`/`method`/`constructor`/`destructor`）。
- 调用点位置：记录 `caller_id`、`callee_id`、`call_file_id`、`call_line`、`call_column`。

### 3.8 函数指针分析 (`ast/function_pointer/`)

**职责**：
- 输入：`CallExpr*`、ASTContext、caller USR。
- 输出：可能的 callee USR 列表（可能为空）。
- 采用逐步增强的策略：
  1. **基础版**：仅分析同一 TU 内对函数指针的直接赋值（如 `p = &func`），构建指针变量到函数的映射。
  2. **进阶版**：结合简单的流不敏感分析，收集函数指针可能的取值（如通过初始化、赋值语句）。
  3. **扩展版**：利用 Clang 的 `Expr::IgnoreImpCasts` 和类型兼容性，枚举所有同类型函数（保守策略）。

**关键实现**：
- 返回的 USR 列表去重。
- 若无法解析，返回空列表（不生成边），或根据配置生成一条指向特殊 USR `"unknown_function_pointer"` 的边，保证调用链完整。
- 与 call_graph 模块通过函数接口耦合，不直接操作 DB。

### 3.9 类关系分析 (`ast/class_relation/`)

**职责**：
- 收集类/结构体定义（symbol 表，kind 为 `class`/`struct`）。
- 收集继承关系（基类列表），生成 `class_relation`（`relation_type='inheritance'`）。
- 收集成员变量（`class_member` 表）。
- 收集类间依赖（如类型引用），生成 `class_relation`（`relation_type='composition'` 或 `'dependency'`）。

**关键实现**：
- 遍历 `CXXRecordDecl`，通过 `getBases()` 获取基类。
- 成员遍历：字段（`FieldDecl`）和成员函数（`CXXMethodDecl`）可选择性记录。
- 依赖关系：当某个成员的类型是另一个类，或函数参数/返回值为类类型时，记录依赖。

### 3.10 数据流分析（全局变量） (`ast/data_flow/`)

**职责**：
- 收集全局变量定义（`global_var` 表）。
- 收集全局变量的读写点，生成 `data_flow_edge`（`var_id`, `accessor_id`（符号ID）, `access_type='read'/'write'`）。

**关键实现**：
- 全局变量判断：`VarDecl` 且具有文件作用域（`isFileVarDecl()`）。
- 读写点：在 `DeclRefExpr` 中，若引用的变量是全局变量，且出现在赋值左侧则为写，否则为读。
- 关联的符号 ID：如果是函数内访问，`accessor_id` 为函数 ID；若在全局初始化中访问，可关联到特殊符号或忽略。

### 3.11 控制流分析 (`ast/control_flow/`)

**职责**：
- 对每个函数构建 CFG（使用 Clang 的 `CFG` 类）。
- 将基本块与边写入 `cfg_node` 和 `cfg_edge` 表。

**关键实现**：
- 为每个函数生成一个 CFG，节点存储基本块起始位置（文件、行号），边存储跳转类型（无条件、条件真、条件假等）。
- 节点内容可包含块内语句的摘要（如第一个语句的行号），供前端展示。
- 关联 `function_id`（symbol ID）。

### 3.12 DB 连接与 Schema (`db/`)

**职责**：
- 封装 SQLite 数据库连接，提供打开、关闭、事务控制接口。
- 创建表结构（与 [数据库设计.md] 一致），并执行必要的迁移。
- 提供对 writer 和 reader 透明的数据库句柄访问。

**关键实现**：
- 打开数据库时设置优化参数：`PRAGMA journal_mode=WAL`、`synchronous=NORMAL`、`cache_size=-64000`、`foreign_keys=ON`。
- 版本管理：创建 `schema_version` 表，记录当前 schema 版本，启动时检查并执行迁移脚本。
- 事务接口：`BeginTransaction()`、`Commit()`、`Rollback()`。

### 3.13 DB 写入 (`db/writer/`)

**职责**：
- 提供批量写入接口：`WriteSymbols`、`WriteCallEdges`、`WriteClasses`、`WriteClassRelations`、`WriteGlobalVars`、`WriteDataFlowEdges`、`WriteCFGNodes`、`WriteCFGEdges` 等。
- 维护 USR→symbol_id 的内存缓存，避免重复查询。
- 维护文件路径→file_id 的内存缓存。
- 使用 UPSERT 语义处理唯一约束冲突（如 symbol 表的 USR 唯一）。

**关键实现**：
- 每个写入函数接收一个记录向量，在单事务中执行插入/替换。
- 缓存更新策略：写入新 symbol 时，同时更新 USR→ID 缓存。
- 对于跨 TU 的符号引用（如 call_edge 的 callee 可能在其他 TU 定义），写入时需确保 callee_id 存在（可能通过缓存获取，若不存在则先插入占位 symbol？但更合理的是在解析阶段已收集所有符号，此处应保证 ID 存在）。
- 为提高性能，可对每个写入函数进行批量参数绑定（使用 `sqlite3_prepare` + 循环绑定）。

### 3.14 DB 读取与查询 (`db/reader/`)

**职责**：
- 提供按 USR、文件、符号名、位置等条件查询 symbol 的接口。
- 提供调用图扩展查询：给定起始符号 ID，向前/向后遍历指定深度，返回节点和边集合。
- 提供类图、数据流、控制流类似查询。
- 支持按文件+行号查询符号（根据 def/decl 位置）。

**关键实现**：
- 深度遍历使用 BFS/DFS，注意环路检测和去重。
- 返回结构体（如 `QueryResult`）包含节点列表和边列表，供 query 层组装 JSON。
- 按位置查询：通过 `def_file_id` 和 `def_line` 范围匹配（允许行号在 [def_line, def_line_end] 内），同时考虑 decl 位置。

### 3.15 查询 JSON 输出 (`query/`)

**职责**：
- 根据查询类型（`call_graph`、`class_graph`、`data_flow`、`control_flow`）调用 `db/reader` 接口，获取结构化数据。
- 将结果组装成 [接口约定.md] §3 规定的 JSON 格式。
- 支持 `--file` + `--line` [--column] 按位置查询符号（`type=symbol_at`），返回该位置涉及的符号列表。

**关键实现**：
- 将 file_id 转换为文件路径（从 `file` 表查询）。
- 处理节点中的 definition/declaration 字段，格式为 `{"file": "path", "line": L, "column": C}`，以及 range 字段。
- 边中携带调用位置信息（若有）。
- 深度默认 3，可通过 `--depth` 覆盖。

### 3.16 公共模块 (`common/`)

**职责**：
- **日志**：支持多级别（INFO, WARNING, ERROR），可输出到文件或 stderr（`--verbose` 控制）。
- **错误码**：定义与 [接口约定.md] §4 一致的错误常量。
- **路径工具**：规范化路径、相对路径转绝对路径、获取工程相对路径。
- **Clang 环境探测**：自动探测系统头文件路径和 Clang 内置资源目录。
  - 执行 `clang++ -E -x c++ - -v`，从 stderr 解析 `#include <...> search starts here:` 至 `End of search list.`，得到系统头路径列表。
  - 执行 `clang++ -print-resource-dir` 得到资源目录。
  - 结果缓存，避免重复探测。
  - 提供 `GetClangIncludeEnv()` 返回包含 `-resource-dir` 和多个 `-isystem` 的字符串向量，供 `ArgumentsAdjuster` 使用。
- **USR 工具**：封装 `clang::index::generateUSRForDecl`，并提供 USR 合法性检查。

## 4. 关键数据流与交互

### 4.1 全量解析流程

1. CLI 解析 `parse` 子命令，构造 `ParseOptions`，调用 `driver::RunParse(opts)`。
2. driver 调用 `compile_commands::Load` 获取全部 TU 列表。
3. 若 `opts.incremental` 为 true，driver 调用 `incremental::GetChangedFiles` 和 `RemoveDataForFiles` 清理旧数据。
4. driver 调用 `history::InsertParseRun` 创建 `parse_run` 记录（`running`）。
5. driver 根据 `opts.lazy` 和 `priority_dirs` 决定首轮 TU 列表（优先目录下的 TU）。
6. driver 将 TU 列表提交给 `scheduler`，并注册进度回调（CLI 层传入）。
7. scheduler 并行执行每个 TU：调用 `combined::RunAllAnalysesOnTU`，得到 `CombinedOutput`，收集到线程安全的结果容器。
8. 所有 TU 完成后，driver 将合并后的 `CombinedOutput` 交给 `db_writer` 批量写入 DB。
9. driver 更新 `parsed_file`（记录每个文件的 mtime/hash）。
10. driver 调用 `history::UpdateParseRun` 标记为 `completed`，记录解析文件数。
11. CLI 收到 100% 进度后，输出解析摘要 JSON（可选项），进程退出。

### 4.2 增量解析流程

同全量解析，但步骤 3 中仅删除变更文件的数据，步骤 5 中仅提交变更文件对应的 TU。

### 4.3 查询（含懒解析）流程

1. CLI 解析 `query` 子命令，调用 `query::RunQuery(opts)`。
2. query 层根据查询类型调用 `db_reader` 对应接口。
3. 若启用了懒解析（`opts.lazy`）且 `db_reader` 发现目标符号/文件尚未解析（通过查询 `parsed_file` 或符号存在性判断），则：
   - 确定需要解析的文件列表（根据 USR 定位定义文件，或根据文件路径）。
   - 调用 `driver::ParseOnDemandForQuery(file_list)`，同步阻塞等待解析完成。
   - 解析完成后，重新调用 `db_reader` 获取最新数据。
4. query 层将结果组装为 JSON，输出到 stdout。

### 4.4 历史列表流程

1. CLI 解析 `list-runs` 子命令，调用 `history::ListRunsJson(opts.db_path, opts.limit)`。
2. history 从 `parse_run` 表查询最近 limit 条记录（按 started_at 降序）。
3. 组装 JSON 数组并输出。

## 5. 关键实现要点与优化策略

### 5.1 单次 AST 遍历合并分析
- 通过 `combined` 模块确保同一 TU 只解析一次。
- 各分析子模块共享同一个 ASTContext，避免重复遍历。
- 错误隔离：每个子模块的 `Analyze` 函数内部捕获所有异常，记录后返回，保证其他模块继续执行。

### 5.2 标准库头文件自动探测与注入
- 探测结果缓存在进程全局，避免重复调用外部进程。
- 通过 `ArgumentsAdjuster` 在 `ClangTool` 运行前注入参数：`-resource-dir` 和多个 `-isystem`。
- 支持用户通过 `extra_include_dirs` 提供额外包含路径。

### 5.3 函数指针分析的分阶段实现
- **第一阶段**：仅分析同一 TU 内的直接赋值语句，构建指针变量到函数的映射表，用于解析该 TU 内的函数指针调用。复杂度低，能满足多数简单场景。
- **第二阶段**：引入跨 TU 的保守策略——类型兼容匹配，将所有与该函数指针类型兼容的函数作为可能目标（通过查询全局 symbol 表）。会生成大量边，但保证不遗漏。
- **第三阶段**：结合数据流分析，实现更精确的跨 TU 指向分析（可选）。

### 5.4 DB 写入性能优化
- 使用 WAL 模式，提高并发写入性能。
- 批量插入：每个写入接口接收向量，在一个事务中批量执行（每 1000 条提交一次，避免大事务）。
- USR→ID 缓存：`std::unordered_map<std::string, int64_t>`，线程安全（读写锁）。
- 文件路径→ID 缓存：类似。
- 使用 `INSERT OR REPLACE` 处理 symbol 表，避免先查后改。

### 5.5 查询深度限制与环路处理
- 遍历时使用 `std::unordered_set` 记录已访问节点，防止无限递归。
- 深度计数：BFS 每层记录距离，超过 `max_depth` 停止扩展。
- 返回的边集和节点集去重，确保 JSON 中无重复。

### 5.6 并发控制与内存管理
- 线程池大小 = 用户指定值或系统核心数。
- 每个 TU 分析独立内存，任务完成后释放。
- 通过任务队列限制同时运行的任务数，防止内存峰值过高。

### 5.7 错误处理与可观测性
- 每个 TU 解析失败时，记录错误日志，并累加失败计数。
- `parse_run` 表增加 `files_failed` 字段，记录失败数量。
- CLI 支持 `--verbose` 输出详细日志到 stderr，便于调试。
- 错误码遵循 [接口约定.md] §4，并在 stderr 输出可读错误信息。

### 5.8 按需解析的同步等待
- `driver::ParseOnDemandForQuery` 返回 `std::future<void>` 或直接阻塞直到完成。
- 内部使用条件变量和完成计数器，等待所有相关 TU 解析完毕。
- 为避免死锁，需确保不会在解析过程中再次触发按需解析（通过标志位防止重入）。

## 6. 模块依赖与接口契约

各模块之间通过明确的接口结构体交换数据，定义在 `common/analysis_output.h` 中。例如：

```cpp
struct SymbolRow {
  std::string usr;
  std::string name;
  std::string kind;  // "function", "class", "variable"...
  int64_t def_file_id;   // 定义所在文件ID
  int def_line;
  int def_column;
  int def_line_end;
  int def_column_end;
  int64_t decl_file_id;  // 声明所在文件ID
  int decl_line;
  int decl_column;
  int decl_line_end;
  int decl_column_end;
  bool def_in_tu_file;   // 定义是否在当前TU主文件
  bool decl_in_tu_file;  // 声明是否在当前TU主文件
};

struct CallEdgeRow {
  int64_t caller_id;
  int64_t callee_id;
  std::string edge_type; // "direct", "via_function_pointer"
  int64_t call_file_id;
  int call_line;
  int call_column;
};
```

其他分析输出类似。这些结构体在模块间传递，最后由 `db_writer` 消费。
