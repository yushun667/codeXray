# 查询结果为空时的诊断步骤

当使用 `query --type symbol_at` 或按「文件 + 行号」解析符号时返回空，可按以下顺序排查。

---

## 1. 确认使用的数据库与参数

- **`--db`** 必须是 **SQLite 数据库文件路径**（例如 `<project>/.codexray/codexray.db`），**不是** `compile_commands.json`。
- 若误将 `compile_commands.json` 传给 `--db`，会报错 `Cannot open DB: ...`（无法作为 SQLite 打开）。
- `compile_commands.json` 仅用于 **parse** 阶段（`--compile-commands`），用于告诉解析器要解析哪些源文件。

**正确示例**：
- 解析：`parse --project /path/to/llvm-project --compile-commands /path/to/llvm-project/build/compile_commands.json`
- 查询：`query --db /path/to/llvm-project/.codexray/codexray.db --project /path/to/llvm-project --type symbol_at --file /path/to/llvm-project/clang/lib/AST/ASTDumper.cpp --line 27`

---

## 2. 确认工程与文件是否在数据库中

在工程根下执行（将 `DB_PATH` 换成实际 `.codexray/codexray.db` 路径）：

```bash
# 查看是否有本工程
sqlite3 DB_PATH "SELECT id, root_path, compile_commands_path FROM project;"

# 查看某文件是否被解析过（路径需与库中一致，一般为绝对路径）
sqlite3 DB_PATH "SELECT id, path FROM file WHERE path LIKE '%ASTDumper%';"
# 或精确路径（与 parse 时写入的路径一致）
sqlite3 DB_PATH "SELECT id FROM file WHERE path = '/Volumes/.../clang/lib/AST/ASTDumper.cpp';"
```

- 若 **project 表为空**：说明从未对该工程执行过 `parse`，或使用了不同的 `--db` 路径。
- 若 **file 表中没有目标文件**：说明该文件**从未被解析**。常见原因是 **该文件不在当时使用的 compile_commands.json 里**。

---

## 3. 为何「文件不在 file 表」？

解析器只处理 **compile_commands.json 中列出的编译单元**。若生成 compile_commands 的构建配置没有包含某个目录或目标，对应源文件就不会出现在 file 表里。

**llvm-project 典型情况**：
- 若 `build/compile_commands.json` 来自只构建了 **llvm**（未构建 **clang**）的配置，则其中通常只有 `llvm/` 下的源文件，**没有** `clang/lib/AST/ASTDumper.cpp` 等 clang 源文件。
- 因此对这些 clang 文件做 symbol_at / call_graph 查询时，会因「文件中不存在于 DB」而得到空结果。

**处理方式**：
1. 使用**包含目标文件**的 compile_commands.json（例如先构建 clang，或使用生成包含 clang 的 compile_commands 的配置）。
2. 对该工程**重新执行 parse**，并指定该 compile_commands：
   ```bash
   codexray-parser parse --project /path/to/llvm-project \
     --compile-commands /path/to/llvm-project/build/compile_commands.json
   ```
3. 解析完成后，再对目标文件执行 query。

---

## 4. 路径一致性（已做规范化）

解析时写入 DB 的路径是**规范化后的绝对路径**（`NormalizePath` / `MakeAbsolute`）。  
自某次更新起，**查询时会对 `--file` 与 `--project` 做同样的规范化**，因此只要传入的路径能解析到同一实际文件，通常能与 DB 中的路径一致。若仍怀疑路径不一致，可用上面第 2 步的 SQL 查看 DB 中实际存储的 `path` 值，与传入的 `--file` 对比。

---

## 5. 小结

| 现象 | 可能原因 | 建议 |
|------|----------|------|
| `Cannot open DB` | `--db` 指向了非 SQLite 文件（如 compile_commands.json） | 使用 `.codexray/codexray.db` 等 SQLite 路径 |
| project 表无记录 | 未对该工程执行 parse 或用了别的 DB | 先执行 parse，并用同一 `--db` 查询 |
| file 表无该文件 | 该文件不在 parse 时使用的 compile_commands 中 | 使用包含该文件的 compile_commands 并重新 parse |
| 有 file 无 symbol | 该行无符号或定义区间不包含该行 | 检查 `symbol.def_line/def_line_end` 与传入行号 |
